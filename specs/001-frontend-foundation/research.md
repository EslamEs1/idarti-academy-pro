# Phase 0 Research: Frontend Foundation

**Feature**: Frontend Foundation, Design System, and App Shell (`001-frontend-foundation`)
**Date**: 2026-05-06

This document resolves the four planning-phase decisions deferred from `/speckit-clarify`, plus a handful of locked design-token decisions that the plan, contracts, and tasks all depend on. Each item follows the Decision / Rationale / Alternatives format.

---

## R1. Tailwind delivery method

**Decision**: TailwindCSS 3.4.x via the official **CLI build** (`npx tailwindcss -i ./assets/css/input.css -o ./assets/css/output.css`). `tailwind.config.js` extends the default theme with the platform's color tokens, typography, and content-glob purge paths covering `index.html`, `pages/**/*.html`, and `components/**/*.html`. The compiled `output.css` is committed to the repo so every page works from `file://` without running `npm install`.

**Rationale**:
- The constitution explicitly permits an npm-based Tailwind toolchain "only for CSS generation" (Principle I). The CLI is the minimum viable toolchain — no PostCSS plugins beyond Autoprefixer, no bundler.
- Committing `output.css` preserves the constitution's "openable from filesystem" guarantee for any reviewer who didn't clone with intent to build.
- A custom theme via `tailwind.config.js` is needed to express the deep-navy / gold / soft-green / dark-slate tokens. Without a config file, Tailwind defaults would be off-brand.
- Tailwind 3.3+ has built-in `rtl:` and `ltr:` modifiers — no third-party RTL plugin required.

**Alternatives considered**:
- **Tailwind Play CDN** (`<script src="https://cdn.tailwindcss.com">`): Eliminates the build step entirely. **Rejected** because the script ships ~3 MB of compiler in-browser, displays a "not for production" warning banner, breaks offline review, and forbids a custom config file via simple inline syntax (would still work but is awkward). For a *premium* prototype shown to stakeholders, the CDN's overhead and warnings are inappropriate.
- **Hand-written CSS only (no Tailwind)**: Would force re-implementation of the spacing scale, color tokens, and responsive utility ergonomics. Higher effort, lower consistency. Rejected.
- **Tailwind via PostCSS-only setup (no CLI)**: Equivalent in output but adds plumbing without value at this scope.

## R2. Icon library

**Decision**: **Lucide icons (v0.x)**, embedded as **inline `<svg>`** snippets directly in each HTML page. No JavaScript icon-injection library. Required icons (Home, Book, Users, GraduationCap, Calendar, Bell, MessageCircle, ChartBar, FileText, CreditCard, Settings, LogOut, Award, ClipboardList, Menu, X, Search, ChevronDown, ChevronLeft, etc.) are copied as static SVG markup into the relevant places.

**Rationale**:
- **Inline SVG is the only icon delivery method that fully satisfies Constitution Principle I** (no runtime JS to render content) AND Principle II (page is self-contained — no external font/icon CDN to break offline).
- Lucide is MIT-licensed, has ~1400 icons including educational, financial, and admin glyphs needed by this UI, and ships with consistent stroke widths that match a "premium, calm, modern" aesthetic.
- Inline SVGs also naturally honor `currentColor`, so badges/buttons inherit Tailwind text colors and `rtl:` modifiers can flip them where needed (e.g., chevrons).

**Alternatives considered**:
- **Heroicons (by the Tailwind team)**: Equally good visually; chose Lucide for slightly broader educational/finance icon coverage (graduation-cap, clipboard-list, wallet, percent). Heroicons remains an acceptable substitute if any specific icon is missing from Lucide.
- **Font Awesome (CDN webfont)**: Rejected — webfont breaks offline, adds 100+ KB, conflicts with constitution's self-contained-page rule.
- **Material Symbols / Material Icons**: Rejected — Google's icon font stylistically reads as Android admin chrome rather than premium education.
- **`lucide` JS injection script**: Rejected — uses JavaScript to render icon content, violating Principle V's "page content MUST NOT be generated by JavaScript".

## R3. Font stack

**Decision**: **Tajawal** (primary, weights 400/500/700/800) loaded from **Google Fonts CDN** with `<link rel="preconnect" href="https://fonts.googleapis.com">` for warm-up. Fallback chain: `'Tajawal', 'Cairo', 'Segoe UI', 'Tahoma', system-ui, sans-serif`. Configured globally in `tailwind.config.js` under `theme.extend.fontFamily.sans`. Line-height utilities tuned for Arabic (`leading-7` on body, `leading-tight` on headings) per FR-031.

**Rationale**:
- Tajawal is purpose-designed for Arabic and Latin co-rendering, widely deployed in Saudi premium digital products (banking, edtech, telecom apps), and reads with confident, modern character at body sizes — matching the constitution's "calm, modern, family-friendly, trustworthy" requirement (Principle IV).
- Google Fonts CDN is conventional, supports `preconnect` for fast first paint, and serves woff2 with broad CDN edge coverage. Pages still open from `file://` — fonts just fall back to system Arabic if offline, which is acceptable for prototype review.
- Cairo as second fallback in case Tajawal is blocked (some corporate networks restrict Google Fonts); both are free, both are well-supported.
- System-Arabic fallback (`Segoe UI` / `Tahoma`) ensures readable rendering on Windows machines without internet.

**Alternatives considered**:
- **Self-host Tajawal woff2 in `assets/fonts/`**: Considered for true offline-strict viewing. Rejected for v1 because (a) the constitution doesn't mandate offline-strict, (b) it adds ~80 KB of binary assets to the repo and per-weight `@font-face` declarations to maintain. Documented as a follow-up enhancement if a later spec demands it.
- **IBM Plex Sans Arabic**: More corporate, slightly less warm. Rejected for tone mismatch.
- **Noto Sans Arabic**: Universal but generic; reads as "default Android system font". Rejected.
- **System Arabic only** (`Segoe UI / Tahoma`): Quickest, but inconsistent across operating systems; loses the premium feel. Rejected.

## R4. Avatar source

**Decision**: **Initial-based avatars** rendered as styled `<div>` elements — circular badge with two-character Arabic initials over a solid color from the role palette. Per role:
- Student → primary navy 700 background, white text (e.g., "ع.م" for عبد الرحمن مؤمن).
- Parent → warm gold 100 background, primary navy 800 text (e.g., "و.أ").
- Teacher → soft green 700 background, white text (e.g., "أ.ب" for أ. أحمد البكري).
- Admin → primary navy 900 background, accent gold 400 text (e.g., "م.م" for مدير المنصة).

Sized at 40×40 px in the header, 56×56 px in dashboard "user welcome" cards.

**Rationale**:
- Fully self-contained: no image files, no CDN, no external request. Aligns with Principle I (static, no APIs) and Principle II (page self-contained).
- Color-per-role doubles as a subtle role identity cue alongside the explicit role label in the header (defense in depth for FR-007 and SC-002).
- Arabic initials feel native to the audience and visually balance the Arabic display copy elsewhere on the page.
- Zero licensing concerns — no need to source human portraits or use stock photography for a prototype.

**Alternatives considered**:
- **Bundled local avatar images** (curated stock or illustration): Rejected — licensing overhead for a prototype, inconsistent style across roles, and risks looking generic ("stock photo dashboard").
- **External avatar service (ui-avatars.com, DiceBear)**: Rejected — external requests break offline review and add a vendor dependency for what is fundamentally a styled `<div>`.
- **Emoji-based avatars**: Rejected — inconsistent rendering across operating systems, unprofessional tone.

## R6. Active sidebar state styling

**Decision**: The active sidebar entry uses three concurrent signals (color + weight + indicator) to satisfy FR-013 and FR-035:

1. Background color: `bg-primary-800` (vs. transparent for inactive).
2. Font weight: `font-semibold` (vs. `font-medium` for inactive).
3. A `border-inline-end-width: 4px` accent-gold (`border-accent-500`) indicator bar. In RTL, `border-inline-end` resolves automatically to the *left* edge of the LI — i.e., the inner edge facing the content area, which is the conventional "active rail" position for RTL sidebars.

**Rationale**: `border-inline-end` is direction-aware — it auto-flips for RTL/LTR, eliminating the need for `rtl:` overrides. Using three signals (background + weight + bar) means the active state survives color-blindness and color-only-rendering edge cases.

**Alternatives considered**: A single accent-gold background — rejected, fails the "color is never the sole signal" rule. A dot icon — rejected, adds asymmetry to the icon column.

## R7. Mobile sidebar drawer behavior

**Decision**: At `< md` breakpoint (`< 768 px`), the sidebar's CSS becomes `position: fixed; inset-block: 0; inset-inline-end: 0; transform: translateX(100%); transition: transform 200ms ease-out;`. A `.is-open` class (toggled by `assets/js/main.js`) sets `transform: translateX(0)`. A semi-transparent backdrop (`<div class="fixed inset-0 bg-black/40">`) fades in beneath the drawer; clicking it or the explicit close button (`✕` in the drawer top-end corner) closes the drawer. A header burger button (`☰`) opens it.

The drawer slides in from the **inline-end edge** (right side under RTL — the natural side for RTL navigation). Body scroll is locked while the drawer is open via `document.body.classList.toggle('overflow-hidden', isOpen)`.

When the viewport crosses to `≥ md`, JS runs a one-time `resize` cleanup that strips the `.is-open` class and re-enables body scroll, preventing the orphaned-overlay edge case called out in spec.md.

**Rationale**: `inset-inline-end` is direction-aware (resolves to right in RTL, left in LTR if a future spec ever enables LTR). Using `transform` (not `display: none`) preserves CSS animations cheaply. Body scroll lock is a standard SaaS UX safeguard.

**Alternatives considered**:
- Full-page overlay menu (no drawer slide): Rejected — feels heavier and less premium.
- CSS `:has()` / sibling selector trick (no JS): Considered, but the close-on-resize requirement (Edge Case in spec.md) genuinely needs JS, so adding ~40 lines of vanilla JS is justified per Principle V.

## R8. Notification & message badge zero state

**Decision**: When unread count = 0, the small numeric badge is **not rendered** (no `0` chip). The icon button itself remains visible and clickable. When count is between 1 and 99, the integer is shown. When count ≥ 100, the badge displays `99+`.

**Rationale**: Standard pattern across premium SaaS dashboards (Slack, Linear, Notion, Tabby). Showing "0" creates needless visual noise and weakens the signaling power of the badge when there *is* an unread item.

**Alternatives considered**: Always showing "0" — rejected for the noise reason. Hiding the icon entirely when count = 0 — rejected, violates user expectation that the notification entry point is always available.

## R9. Page templating approach (header & sidebar reuse)

**Decision**: A `components/` folder at repo root contains canonical header and per-role sidebar HTML files. These files are **reference partials**, not server-side includes or JS-injected fragments. Each delivered page (`index.html`, the four dashboards, `pages/ui-kit.html`) embeds a hand-copied instance of the relevant header and sidebar. Updates to the canonical partial require copying the change into every page.

A short authoring note at the top of each `components/*.html` file marks it as "REFERENCE — DO NOT LOAD AT RUNTIME" so future contributors don't try to wire it up dynamically.

**Rationale**: This is the *only* approach that simultaneously satisfies (a) Constitution Principle II's "every page is fully self-contained, openable as static HTML" and (b) the practical need to keep the header/sidebar visually consistent across pages without re-typing them from scratch.

The cost is real — when the header changes, all five in-app pages must be re-edited. For six pages this is acceptable; the alternative (a build-time include step) would introduce a transformation pipeline beyond what Tailwind's CLI does, exceeding the constitution's "TailwindCSS only for CSS generation" tolerance.

**Alternatives considered**:
- HTML `<iframe>`-injected header/sidebar: Rejected — breaks RTL containment, adds focus/scroll bugs, looks unprofessional.
- Server-side includes (SSI / Eleventy / 11ty / a static site generator): Rejected — adds a runtime/build pipeline beyond Tailwind, violates constitution Principle I.
- JS `fetch()` of a partial and `innerHTML` injection: Rejected — uses JS to render content (banned by Principle V).

## R10. Build & dependency footprint

**Decision**: `package.json` has exactly these devDependencies:
- `tailwindcss@^3.4.0`
- `@tailwindcss/forms@^0.5.7`
- `autoprefixer@^10.4.0` (PostCSS, runs in Tailwind's pipeline automatically)

And exactly these scripts:
- `"build:css": "tailwindcss -i ./assets/css/input.css -o ./assets/css/output.css --minify"`
- `"watch:css": "tailwindcss -i ./assets/css/input.css -o ./assets/css/output.css --watch"`

No bundler, no linter, no test runner, no formatter. `node_modules/` is gitignored; `assets/css/output.css` is **committed** so reviewers can open pages without `npm install`.

**Rationale**: Smallest possible footprint that still satisfies the constitution and gives us the custom theme. Anything more (eslint, prettier, vitest, vite) is yagni for a six-page prototype.

**Alternatives considered**: Add ESLint for the ~80-line `main.js` — rejected, overkill. Add `htmlhint` for the static pages — rejected, the manual visual review at the spec's success-criteria stage is sufficient validation.
