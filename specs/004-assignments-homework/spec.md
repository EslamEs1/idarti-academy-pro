# Feature Specification: Assignments and Homework Frontend

**Feature Branch**: `004-assignments-homework`
**Created**: 2026-05-08
**Status**: Draft
**Input**: User description: "Build a complete homework experience that proves the academy follows up seriously. Students must clearly see required homework, submission status, teacher feedback, resubmission requests, Quran-specific homework, and progress."

## Overview

This feature adds the **Assignments and Homework** layer that the إدارتي academy promises on every other page but has not yet rendered. The student-side dashboard, weekly plan, learning journey, my-courses, course-details, and live-session-details pages already link to `assignments.html`; the teacher dashboard already links to `homework-review.html`. Spec 004 builds the six pages those links point to so the homework narrative — assigned → submitted → reviewed → graded → (revised) → accepted — is fully visible end-to-end without a backend.

The spec introduces six new linked pages — four on the student side, two on the teacher side — and establishes the Quran-specific evaluation pattern (memorization / tajweed / pronunciation / fluency scoring) that distinguishes a Quran-academy homework system from a generic LMS:

1. `pages/student/assignments.html` — the واجباتي index: summary cards, status filters, course/teacher/month filters, and a list/table of every assignment with status, grade, and CTAs.
2. `pages/student/assignment-details.html` — a single assignment's full brief: task, instructions, attached materials (PDF/audio/video/image/external link), Quran-specific brief (ayah range, tajweed focus, common mistakes), submission history, and Submit/Resubmit CTA.
3. `pages/student/submit-assignment.html` — the submission surface: text answer, file/audio/image/video upload, external link, completion confirmation, student-to-teacher note, submission checklist, visual-only Submit button.
4. `pages/student/submission-feedback.html` — the post-submission feedback surface: status, teacher feedback, grade, strengths/improvements, Quran evaluation (memorization/tajweed/pronunciation/fluency/mistakes count/ayahs needing review), resubmission request, attempts remaining, Resubmit CTA.
5. `pages/teacher/homework-review.html` — the teacher's grading queue: summary cards, course/student/status/date filters, a submissions table with Student/Assignment/Course/Submitted-at/Status/Attempts and a Review CTA.
6. `pages/teacher/homework-submission-details.html` — the teacher's grading surface: student info, assignment info, submitted files visual list, student notes, grading form (grade, feedback, strengths, improvements, status selector, request-resubmission checkbox), Quran evaluation form (memorization/tajweed/pronunciation/fluency scores + notes), visual-only Save Review button.

The feature is purely visual / frontend. No real upload, no real grading persistence, no API calls — every status pill, every grade number, every Save button is static markup the user can read but not act against. The point of the build is to make **three propositions** unmistakable on the screen:

1. **The academy follows up on every assignment** — required vs submitted vs under review vs accepted vs needs-revision vs late vs rejected are all distinct, named states with their own rows.
2. **Quran homework is graded differently** — a four-axis evaluation (memorization / tajweed / pronunciation / fluency) plus a mistakes count and ayahs-to-review list shows up wherever a Quran assignment appears, on both the student feedback and the teacher grading surfaces.
3. **The resubmission cycle is real** — when work is returned, the student sees teacher feedback, attempts remaining, and a Resubmit CTA; the teacher's grading form has an explicit "request resubmission" checkbox and a status selector that includes "needs revision".

These three propositions × the existing Abdulrahman persona (Arabic student, Quran path level 3, family balance 750 ر.س) anchor every sample row on every page. The dashboard's claim of "3 واجبات مستحقة" must reconcile exactly with the assignments index's "واجبات مطلوبة" summary card.

## Clarifications

### Session 2026-05-08

- Q: How should reviewers read the relationship between the four Quran-axis scores (الحفظ 85, التجويد 60, النطق 75, الطلاقة 70 — average ≈ 72.5) and the final grade (65/100) on `submission-feedback.html`? → A: **Independent** — axes are diagnostic per-skill scores; final grade is a separate holistic teacher judgment. A one-line caption on `submission-feedback.html` and `homework-submission-details.html` MUST explicitly explain that the overall grade is the teacher's holistic assessment and the axes are diagnostic detail (so the 72.5 vs 65 difference reads as intentional, not as a math error). The same caption appears symmetrically on the teacher's grading form.
- Q: How should the two teacher-only status options ("مقبول مع ملاحظات" and "تأجيل المراجعة") surface on the student side, given the student-side filter chips don't include them? → A: **Collapse to existing student-side values.** "مقبول مع ملاحظات" maps to "مقبول" as the student-facing status pill (the notes themselves appear inside the feedback panel — strengths/improvements lists + ملاحظة المعلم quote — not as a separate status). "تأجيل المراجعة" maps to "قيد المراجعة" on the student side (deferment is teacher-internal; the student continues to see "awaiting review"). No new student-side filter chips, pills, or list-row variants are added. The teacher's grading form retains all five options (FR-063); the student-side taxonomy retains its existing six row-status values (FR-014). The mapping is documented in Assumptions and enforced by a new FR.
- Q: What's the meaningful distinction between the "جديد" and "مطلوب التسليم" status filters on `assignments.html`? → A: **Recency vs deadline.** "جديد" tags assignments posted within the last 48 hours that the student hasn't acted on yet (visually accented as a notification-style chip — a "fresh" notice). "مطلوب التسليم" tags assignments posted earlier than 48 hours, with the due date approaching, not yet submitted (visually a standard actionable chip — a "deadline" reminder). A row carries exactly one of the two pills at any time; once a "جديد" row crosses the 48-hour age boundary, it becomes "مطلوب التسليم". On the assignments-index sample data, at least one row uses each pill — the lesson-recap homework (assigned 7 مايو 2026, due 9 مايو 2026) is "جديد"؛ the سورة الفجر memorization (assigned 4 مايو 2026, due 12 مايو 2026) is "مطلوب التسليم". The "واجبات مطلوبة (3)" summary card counts BOTH "جديد" and "مطلوب التسليم" rows since both demand student action.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Reviewing My Assignments Index (Priority: P1)

A student opens the **Assignments** page from the sidebar (or the dashboard's "عرض كل الواجبات" link, or any "عرض الواجب" link from the weekly plan / dashboard / live-session-details page). They want a single-glance view of: how many assignments are required of them right now, how many they've already submitted, what's awaiting teacher review, what was returned for revision, and what is overdue — and they want to be able to filter the list by status / course / teacher / month.

**Why this priority**: This is the canonical homework surface — every other surface in the platform points to it. Without it, none of the dashboard / weekly-plan / live-session "عرض الواجب" links resolve to a real destination. It also establishes the status taxonomy that the rest of the feature builds on; if the index doesn't render every status (جديد / مطلوب التسليم / تم التسليم / قيد المراجعة / مقبول / يحتاج تعديل / متأخر / مرفوض), no downstream page recovers that breadth.

**Independent Test**: Open `pages/student/assignments.html` from the file explorer. Without any other context, the viewer must be able to answer: "How many assignments require my action right now? How many have I already submitted? Which one is overdue? Which was returned for revision? Who teaches each? Which course does each belong to? What grade did I get on the accepted ones?" — entirely by reading the page. Filter chips, course/teacher/month dropdowns, and the search input must be visually present and obvious, even though they are non-functional in this prototype.

**Acceptance Scenarios**:

1. **Given** a student opens the assignments page, **When** they look at the page header, **Then** they see the title "واجباتي" (h1), a one-line subtitle that frames the page (e.g., "تابع كل واجباتك في مكان واحد — مطلوب، مسلَّم، تحت المراجعة"), and the user persona's name in the contextual greeting reusing the dashboard's chrome.
2. **Given** the student wants a glance at workload, **When** they look below the header, **Then** they see exactly five summary cards stacked horizontally on desktop / two-up on tablet / one-up on mobile: "واجبات مطلوبة" (3), "تم التسليم" (4), "بانتظار المراجعة" (1), "تحتاج تعديل" (1), "متأخرة" (1) — each card uses a distinct accent color paired with an icon, never color alone.
3. **Given** the student wants to narrow the list, **When** they look at the filter row, **Then** they see at least nine status filter chips (الكل، جديد، مطلوب التسليم، تم التسليم، قيد المراجعة، مقبول، يحتاج تعديل، متأخر، مرفوض) — "الكل" is selected by default — plus four secondary filter dropdowns (الدورة، المعلم، الشهر، الحالة) above the table; all are visual-only.
4. **Given** the assignment list shows ≥ 8 rows, **When** the student looks at any row, **Then** they see: assignment title, course chip, teacher chip with avatar, due date (with relative caption like "بعد 4 أيام" or "متأخر بـ 6 أيام"), status pill drawn from the catalog, grade column ("—" for un-graded, "92 / 100" for accepted), and a CTA cluster ("عرض التفاصيل" always; "تسليم" only when the row is actionable).
5. **Given** the student is on a mobile viewport, **When** they look at the same list, **Then** rows collapse to single-column cards that preserve title / course / teacher / due date / status / grade / CTAs without horizontal page scroll.
6. **Given** the student wants to find one specific assignment, **When** they look near the top of the page, **Then** they see a search input with a placeholder hint (e.g., "ابحث عن واجب أو دورة...") — purely visual but obviously present.

### User Story 2 — Drilling into an Assignment (Priority: P1)

A student taps "عرض التفاصيل" on any row of the assignments index, or "عرض الواجب" from the dashboard / weekly plan / live-session-details, and lands on the **Assignment Details** page. They want a complete picture of the assignment: who set it, when, when it's due, what state it's currently in, what they're being asked to do, what teacher instructions apply, what supporting materials are attached, the Quran-specific brief if it's a Quran assignment, the submission history, and a clear primary CTA — Submit if they haven't submitted yet, Resubmit if it was returned for revision.

**Why this priority**: This page is the only place in the prototype that explains *what* a single assignment is. Every other surface only shows status / grade / CTAs. Without this page, the student-side homework flow is incomplete and the resubmission narrative has nowhere to anchor.

**Independent Test**: Open `pages/student/assignment-details.html` from the file explorer. A viewer must be able to answer: "What is this assignment? What ayah range am I being asked to memorize? What tajweed focus did the teacher highlight? Which common mistakes should I avoid? When was this assigned and when is it due? What materials did the teacher attach? Have I submitted before, and what happened? What's my next action?" — entirely by reading the page.

**Acceptance Scenarios**:

1. **Given** the student opens assignment-details, **When** they read the top of the page, **Then** they see: assignment title (h1) — "تطبيق أحكام الميم الساكنة من سورة البقرة (الآيات 1-10)" — a meta strip with the course chip ("القرآن الكريم — المستوى الثالث"), the teacher chip ("الأستاذ أحمد العمري"), the creation date ("أُنشئ في 2 مايو 2026"), the due date ("التسليم قبل 10 مايو 2026"), the current status pill ("يحتاج تعديل"), and the prior grade ("65 / 100").
2. **Given** the student wants to know what to do, **When** they look below the meta strip, **Then** they see a "المطلوب منك" section with a 2-3 sentence task statement followed by a numbered list of ≥ 3 sub-tasks (e.g., "احفظ الآيات 1 إلى 10 من سورة البقرة"، "سجّل تلاوتك بصوت واضح"، "طبّق أحكام الميم الساكنة في كل موضع").
3. **Given** the student wants the teacher's guidance, **When** they look further down, **Then** they see a "تعليمات المعلم" panel with ≥ 4 sentences from الأستاذ أحمد including encouragement, the specific tajweed rules to apply, advice on recording quality, and a sentence about resubmission expectations.
4. **Given** the assignment includes attached materials, **When** the student looks at the "الملفات والمراجع المرفقة" section, **Then** they see ≥ 5 attachment tiles representing each material type at least once: a PDF tile ("ملخص أحكام الميم الساكنة.pdf"), an audio tile ("نموذج تلاوة من الشيخ المنشاوي.mp3"), a video tile ("شرح الإخفاء الشفوي.mp4"), an image tile ("جدول أحكام الميم الساكنة.png"), and an external link tile ("تطبيقات على Quran.com" with a real `https://quran.com/2` href). Each tile shows file type, name, and size or duration.
5. **Given** the assignment is Quran-specific, **When** the student looks at the "البريف القرآني" section, **Then** they see four labeled blocks: نطاق الحفظ والمراجعة (sura name + ayah range + word count), تركيز التجويد (the specific rule, e.g., "أحكام الميم الساكنة: الإخفاء الشفوي، الإدغام الشفوي، الإظهار الشفوي"), أخطاء شائعة يجب تجنّبها (a bulleted list of ≥ 3 common mistakes), and طريقة التسجيل المطلوبة (a one-line guidance such as "مكان هادئ، 60-90 ثانية، صوت واضح").
6. **Given** the student has submitted before, **When** they look at the "سجل التسليمات" section, **Then** they see a vertical timeline with ≥ 1 prior attempt entry showing: attempt number ("المحاولة 1 من 3 محاولات مسموح بها"), submission date/time, the file submitted, the teacher's verdict ("إعادة التسليم مطلوبة")، the grade given on that attempt (65/100)، and a "عرض ملاحظات المعلم" link to `submission-feedback.html`.
7. **Given** the assignment is in the "needs revision" state with attempts remaining, **When** the student looks at the action area, **Then** they see a primary "إعادة التسليم" CTA (links to `submit-assignment.html`), a secondary "تنزيل ملفات الواجب" link, and an "العودة لقائمة الواجبات" tertiary link.

### User Story 3 — Submitting an Assignment (Priority: P1)

A student taps "تسليم" on the assignments index OR "إعادة التسليم" on the assignment-details page and lands on the **Submit Assignment** page. They want one clear surface to upload their work using whichever method matches the assignment (text answer, file, audio recording, image, video, external link, or simply confirming completion for non-deliverable tasks), to leave a short note for the teacher, to walk through a checklist before submitting, and to tap a clearly visible Submit button.

**Why this priority**: This is the action surface — every assigned-state assignment funnels here. Without it, the homework loop has no entry to the "تم التسليم" state. The prototype must make every supported submission method visible so the academy's flexibility (typed essay, recited audio, photographed worksheet, recorded video, external link to a coding sandbox, or just "I did it") is unmistakable.

**Independent Test**: Open `pages/student/submit-assignment.html` from the file explorer. A viewer must be able to answer: "Which assignment am I submitting? What submission methods can I use? What checks should I perform before submitting? Where do I leave a note for the teacher? Where is the Submit button?" — entirely by reading the page.

**Acceptance Scenarios**:

1. **Given** the student opens submit-assignment, **When** they read the top of the page, **Then** they see a "ملخص الواجب" card pinning the assignment title, course, teacher, due date, attempt number ("المحاولة 2 من 3"), and a back link to assignment-details.
2. **Given** the student wants to choose a submission method, **When** they look at the "طريقة التسليم" section, **Then** they see ≥ 7 distinct method blocks each with its own header, icon, helper text, and visual UI: نص الإجابة (a multi-line textarea with placeholder), رفع ملف (drag-and-drop zone showing accepted file types and max size), رفع تسجيل صوتي (a drop zone + a "تسجيل مباشر" stub button), رفع صورة (a drop zone + camera icon), رفع فيديو (a drop zone + max-duration hint), رابط خارجي (a URL input + helper text), and تأكيد الإنجاز (a labeled checkbox for non-deliverable tasks). Methods that don't apply to this assignment are visually muted with a small "غير مطلوب لهذا الواجب" caption — methods are never hidden.
3. **Given** the assignment is a Quran recitation submission, **When** the student looks at the audio block, **Then** the audio block is the prominent / pre-expanded one, with a playback waveform placeholder, a "تسجيل لمدة 60-90 ثانية" hint, a sample mock filename ("تسجيل-الميم-الساكنة-المحاولة-2.mp3") in the upload preview area, and a duration indicator ("01:14 / 01:30").
4. **Given** the student wants to add a note for the teacher, **When** they look below the method blocks, **Then** they see a "ملاحظة للمعلم" textarea with a placeholder hint and a small caption ("سيراها المعلم مع تسليمك").
5. **Given** the student wants to verify before submitting, **When** they look at the "قبل التسليم — قائمة التحقق" section, **Then** they see a checklist of ≥ 5 items each with a labeled checkbox: "راجعت الآيات قبل التسجيل"، "التسجيل في مكان هادئ بدون تشويش"، "صوتي واضح وغير مكتوم"، "طبّقت أحكام الميم الساكنة"، "استمعت لتسجيلي قبل الرفع". The first 4 are pre-checked; the 5th is unchecked to demonstrate both states.
6. **Given** the student is ready, **When** they look at the page footer, **Then** they see a primary "تسليم الواجب" button (visual `<button type="button">` — no form action), a secondary "حفظ كمسودة" link, and a tertiary "إلغاء" link back to assignment-details. A small inline notice clarifies that this is a visual prototype and no upload is processed.

### User Story 4 — Reading Feedback and Resubmitting (Priority: P2)

A student taps the "عرض ملاحظات المعلم" link from the assignments index OR from the submission history on assignment-details and lands on the **Submission Feedback** page. They want to see exactly what the teacher said, what grade they got, what they did well, what they need to improve, the Quran-specific evaluation breakdown if it's a Quran assignment, whether resubmission is required, how many attempts they have left, and a clear Resubmit CTA when applicable.

**Why this priority**: This is the closing surface of the homework loop — and the surface that proves the academy's "we follow up seriously" promise. Without it, the dashboard's "آخر ملاحظة من المعلم" preview has nowhere to expand into. P2 rather than P1 because the action (resubmit) is also reachable directly from assignment-details; this page deepens the explanation rather than gating the loop.

**Independent Test**: Open `pages/student/submission-feedback.html` from the file explorer. A viewer must be able to answer: "What did the teacher say about my submission? What grade did I get? What did I do well? What do I need to fix? On the Quran scoring axes, where am I strong and where am I weak? Do I need to resubmit, and how many attempts do I have left?" — entirely by reading the page.

**Acceptance Scenarios**:

1. **Given** the student opens submission-feedback, **When** they look at the page header, **Then** they see the assignment title, course chip, teacher chip, the submission's status pill ("يحتاج تعديل")، the grade ("65 / 100") rendered as both a numeric score and a small visual progress bar, and the date/time the teacher graded it.
2. **Given** the student wants the teacher's voice, **When** they look at the "ملاحظة المعلم" section, **Then** they see a quote block with a 4-6 sentence Arabic feedback message from الأستاذ أحمد referencing specific moments in the recitation, a small teacher avatar + name caption beneath, and the date the feedback was authored.
3. **Given** the student wants the strengths/improvements breakdown, **When** they look further down, **Then** they see two side-by-side panels (stacked on mobile): "نقاط قوّتك" with ≥ 3 green-accented bullet points each preceded by a check-circle icon, and "نقاط للتحسين" with ≥ 3 amber-accented bullet points each preceded by an alert-circle icon. Color is paired with text and icon — never alone.
4. **Given** the assignment is Quran-specific, **When** the student looks at the "تقييم الأداء القرآني" section, **Then** they see a labeled grid with four scored axes — الحفظ (85/100)، التجويد (60/100)، النطق (75/100)، الطلاقة (70/100) — each with a numeric score, a horizontal bar, and a one-line teacher comment. Below the grid, two summary lines: "عدد الأخطاء المرصودة: 5 أخطاء" and "آيات تحتاج إلى مراجعة: الآيات 3 و 4 و 7".
5. **Given** the submission was returned for revision, **When** the student looks at the resubmission section, **Then** they see a "طلب إعادة التسليم" panel with: a clear "المعلم طلب إعادة تسليم هذا الواجب" headline, the resubmission deadline (10 مايو 2026), an attempts-remaining indicator ("لديك محاولتان متبقيتان من أصل 3")، and a primary "إعادة التسليم الآن" CTA linking to `submit-assignment.html`.
6. **Given** the student wants to compare attempts, **When** they look at the bottom of the page, **Then** they see a "محاولاتك السابقة" timeline with the current attempt expanded and any prior attempts collapsed but listed (date, file name, grade, status pill); for the prototype this section shows attempt 1 (the current one) plus an annotation explaining how additional attempts will appear after resubmission.

### User Story 5 — Reviewing the Homework Queue (Priority: P1)

A teacher opens the **Homework Review** page from the teacher sidebar (or the teacher dashboard's "فتح قائمة التصحيح" link). They want a single workspace that tells them how many submissions are awaiting their review right now, how many they've already accepted today, how many they've returned for revision, how many came in late, and a sortable / filterable table of every submission ready to be opened and graded.

**Why this priority**: This is the teacher's primary daily surface for the homework feature — the equivalent of the student's assignments index. Without it, the teacher dashboard's CTA has no destination and the teacher-side flow has no entry point. P1 because the brief explicitly requires "Teacher review flow is complete" as an acceptance criterion.

**Independent Test**: Open `pages/teacher/homework-review.html` from the file explorer. A viewer must be able to answer: "How many submissions are pending my review right now? How many did I accept today? Which student submitted what, when, and is this their first attempt or a resubmission? What's the next submission I should grade?" — entirely by reading the page.

**Acceptance Scenarios**:

1. **Given** the teacher opens homework-review, **When** they look at the page header, **Then** they see the title "مراجعة الواجبات" (h1), a contextual greeting reusing the teacher dashboard chrome ("الأستاذ أحمد"), and a one-line subtitle ("راجع تسليمات طلابك وقدّم ملاحظات تربوية واضحة").
2. **Given** the teacher wants a workload glance, **When** they look below the header, **Then** they see exactly four summary cards: "بانتظار المراجعة" (4)، "تم القبول اليوم" (2)، "تحتاج تعديل" (1)، "تسليمات متأخرة" (1) — each with a distinct accent color paired with an icon, never color alone.
3. **Given** the teacher wants to narrow the queue, **When** they look at the filter row, **Then** they see at least four secondary filter dropdowns above the table — الدورة, الطالب, الحالة, التاريخ — and a search input ("ابحث باسم الطالب أو الواجب...") — all visual-only.
4. **Given** the queue shows ≥ 8 submission rows, **When** the teacher looks at any row, **Then** they see: student avatar + name + grade-level chip, assignment title, course chip, submitted-at column (date + time + relative caption like "قبل ساعتين")، status pill, attempts column ("المحاولة 1 من 3" or "المحاولة 2 من 3 (إعادة تسليم)"), and a primary "مراجعة" CTA linking to `homework-submission-details.html`.
5. **Given** at least one row is the persona's pending submission, **When** the teacher looks at the queue, **Then** the first or second row is "عبدالرحمن السعد" with the assignment "تلاوة سورة الضحى مع تطبيق أحكام التجويد"، course "القرآن الكريم — المستوى الثالث"، submitted today, status "بانتظار المراجعة"، attempts "المحاولة 1 من 3" — so the student-side and teacher-side narratives reconcile: this is the same submission the student listed as "قيد المراجعة" on `assignments.html`.
6. **Given** the teacher is on a mobile viewport, **When** they look at the queue, **Then** rows collapse to single-column cards that preserve student / assignment / course / submitted-at / status / attempts / Review-CTA without horizontal page scroll.

### User Story 6 — Grading a Submission (Priority: P1)

A teacher taps "مراجعة" on any homework-review row and lands on the **Submission Details** page. They want a focused workspace that shows them the student, the assignment context, the actual files the student submitted (audio playable, files listed, images previewable), any note the student wrote, and the full grading form: numeric grade, written feedback, strengths, improvements, status selector, request-resubmission checkbox, and — for Quran assignments — the four-axis Quran evaluation form (memorization / tajweed / pronunciation / fluency scores plus notes), then a single Save button.

**Why this priority**: This is the teacher's only authoring surface in the homework loop — every grade the student sees originates here. Without it, the "Teacher review flow is complete" acceptance criterion fails. The Quran evaluation form is the centerpiece that distinguishes this page from a generic LMS grading screen.

**Independent Test**: Open `pages/teacher/homework-submission-details.html` from the file explorer. A viewer must be able to answer: "Which student submitted this and what's their level? Which assignment is this and how is it specified? What did the student actually submit? Did they leave a note? Where do I enter the numeric grade? Where do I write feedback? Where do I list strengths and improvements? Where do I score the four Quran axes? How do I mark this as 'needs revision' and request resubmission? Where is the Save button?" — entirely by reading the page.

**Acceptance Scenarios**:

1. **Given** the teacher opens submission-details, **When** they read the top of the page, **Then** they see a two-column layout (single column on mobile): the **left column** shows the read-only context — student card (avatar, name "عبدالرحمن السعد"، grade level "المستوى الثالث"، parent name, total assignments completed), assignment card (title, course, due date, attempt number "المحاولة 1 من 3"), submitted files list (audio file with play button + duration, ≥ 1 supplementary file), and the student's note quote block — and the **right column** holds the grading form.

### Edge Cases

The following states are documented as edge-case patterns. Where the prototype renders one variant, the alternative variant is documented inline (annotation or comment) so the designer's intent is preserved without doubling the page count.

- **No assignments yet (empty assignments index)**: The page MUST document an empty state — a centered illustration placeholder, copy "لا توجد واجبات حالياً — استمتع بوقتك أو راجع دروسك السابقة"، and a CTA back to my-courses. The prototype itself renders the *populated* state; the empty-state copy lives in a comment block at the bottom of the file for documentation only.
- **All filters return zero rows**: If a student picks a status chip that has no matching rows (e.g., "مرفوض" when nothing is rejected), the table area MUST render an inline empty-state strip ("لا تطابقات لهذا التصفية — جرّب تصفية أخرى") with a "إعادة الضبط" link. Documented in the same comment block as above.
- **Submission with no attached materials**: The assignment-details "الملفات والمراجع المرفقة" section shows an empty-state caption "لم يرفق المعلم ملفات لهذا الواجب — التعليمات النصية كافية" — documented inline; prototype renders the populated variant.
- **No prior attempts (first submission)**: The submission-history timeline on assignment-details shows a single "لا توجد محاولات سابقة — هذا أول تسليم لك" caption — documented inline; prototype renders the resubmission variant where ≥ 1 prior attempt exists.
- **Grade not yet given (under review)**: The grade column on the assignments index shows "—" instead of a number; on the assignment-details page the grade meta-strip cell shows "بانتظار التقييم". Both are exercised on the prototype — at least one row in the index has "—" to demonstrate this.
- **No attempts remaining (rejected)**: When a submission is rejected and no further attempts are allowed, the submission-feedback page MUST show the resubmission panel in a disabled-looking variant: "لا يمكنك إعادة التسليم — الواجب مرفوض من قِبل المعلم" with no Resubmit CTA. Documented inline; prototype renders the "needs revision with attempts remaining" variant on submission-feedback.html.
- **Late submission attempt window closed**: When a late assignment also has a hard close-date past today, the assignment-details action area MUST show "انتهت فترة التسليم — راجع المعلم لمعرفة الخيارات" with no Submit CTA. Documented in the assignments-index late-row caption.
- **Non-Quran assignment**: When an assignment is not Quran-specific (e.g., a math worksheet, an Arabic essay), the Quran-specific brief on assignment-details and the Quran evaluation panel on submission-feedback / submission-details MUST be omitted — not blanked out, not rendered with empty values. The prototype demonstrates the *Quran* variant; non-Quran variants are documented inline. The assignments-index list MUST contain ≥ 2 non-Quran rows so the variety is visible at the index level.
- **Saved-draft state**: The submit-assignment page documents that "حفظ كمسودة" would persist across visits without submitting; the prototype's Submit and Draft buttons are both visual stubs.

## Requirements *(mandatory)*

### Functional Requirements

The functional requirements below are organized first by cross-page items, then by page (matching the user-story groupings), then by cross-page consistency rules. Every FR is testable by inspecting the rendered HTML; none requires runtime interaction.

#### Cross-page items (apply to all six pages)

- **FR-001**: Each new page MUST reuse the Spec 001 / Spec 002 page chrome — same desktop sidebar (student or teacher variant), same top header with the search/notifications/profile area, same Tajawal font, same color tokens, same `dir="rtl"` root, same Lucide-as-inline-SVG pattern.
- **FR-002**: The student sidebar's "الواجبات" entry (already linking to `assignments.html` from Spec 001/002) MUST be marked active on all four student pages; the teacher sidebar's "مراجعة الواجبات" entry MUST be marked active on both teacher pages.
- **FR-003**: Every CTA on every new page MUST link to a real existing HTML file — either an in-prototype page (any page from Spec 001/002/003 or the six pages of this Spec) or to a planned-future page that uses a placeholder anchor `#planned` documented in a comment. No `href="#"` is permitted.
- **FR-004**: Each page MUST be openable directly from the file system without a dev server. No script tag may load remote dependencies beyond the Google Fonts CSS that prior pages already load.
- **FR-005**: No new JavaScript handlers may be added in `assets/js/main.js`. The only sanctioned interactions are the existing Spec 001 sidebar drawer / dropdown / modal / `data-tabs` ARIA-tablist handler. If tab UI is needed (none of the six pages currently requires it), reuse the existing handler with no additions.
- **FR-006**: Every page MUST be responsive at viewport widths 375 / 768 / 1024 / 1280 px with no horizontal page-level scrollbar at any breakpoint.

No new student-side pill variants are introduced for these two teacher outcomes.

- **FR-076**: No real `href="#"` placeholder links may exist on any of the six pages. - **FR-079**: Every form input on the four interactive surfaces (submit-assignment, submission-details grading form) MUST have a visible label tied via `for`/`id`؛ icons MAY supplement but MUST NOT replace text labels (Constitution Information Architecture).

### Key Entities *(visual / sample data)*

The feature uses static sample data anchored to the Spec 001/002/003 personas. None of these entities are persisted, fetched, or rendered by JavaScript — every value is committed in HTML.

- **Assignment** — A required learning task. Attributes: title (Arabic), course reference, teacher reference, creation date, due date, type (Quran / Arabic / English / Math / Islamic Studies / general), Quran-specific flag, attached materials (list of {type: PDF/audio/video/image/external-link, name, size-or-duration, optional href}), task statement, sub-task list, teacher instructions text, Quran brief (sura, ayah range, tajweed focus, common-mistakes list, recording guidance) when Quran-specific, attempts allowed (sample: 3), current attempts used, current status (drawn from the badge catalog), current grade, resubmission deadline.
- **Submission** — A single attempt by a student against an assignment. Attributes: assignment reference, student reference, attempt number, submitted-at timestamp, submission methods used (text / file / audio / image / video / external-link / completion-confirmation), submitted artifacts (list of files or text or URL), student note to teacher, teacher verdict (status), grade, teacher feedback text, strengths list, improvements list, Quran evaluation when Quran-specific (memorization 0-100, tajweed 0-100, pronunciation 0-100, fluency 0-100, mistakes count, ayahs needing review).
- **Quran Evaluation** — The four-axis evaluation pattern unique to Quran assignments. Attributes: memorization score, tajweed score, pronunciation score, fluency score, mistakes count (integer), ayahs needing review (list of ayah references), per-axis teacher comment, additional recitation notes. This entity appears identically (read-only) on the student's submission-feedback page and (editable) on the teacher's submission-details page.
- **Submission Method** — A way the student can submit work. Values: text-answer, file-upload, audio-upload, image-upload, video-upload, external-link, completion-confirmation. Each has its own UI block on the submit-assignment page even when not applicable to the current assignment (muted, never hidden).
- **Resubmission Request** — A teacher decision returning a submission for revision. Attributes: parent submission reference, deadline, attempts remaining, teacher feedback summary. Drives the "إعادة التسليم" CTA visibility on assignment-details and submission-feedback.
- **Status** — The lifecycle state of an assignment (from the student's perspective) or a submission (from the teacher's perspective). Values for student-side assignment: جديد، مطلوب التسليم، تم التسليم، قيد المراجعة، مقبول، يحتاج تعديل، متأخر، مرفوض. Values for teacher-side grading-form selector: مقبول، مقبول مع ملاحظات، يحتاج تعديل، مرفوض، تأجيل المراجعة. All values are drawn from the existing badge catalog; no new variants are introduced.
- **Teacher Grading Form Snapshot** — The composed editable state on `homework-submission-details.html`. Attributes: numeric grade (0-100), written feedback, strengths list, improvements list, status selection, request-resubmission flag, Quran evaluation block (when applicable), per-axis comments. Values are pre-filled to the persona's narrative state (grade 65, status "يحتاج تعديل"، resubmission requested).
- **Attempts Counter** — An integer pair (used / allowed) shown on assignment-details, submit-assignment, submission-feedback, and the teacher's submission-details. Anchored to the persona's state: 1 used, 3 allowed for the featured assignment after the first return; 0 used, 3 allowed for the upcoming "تلاوة سورة الضحى" submission viewed on the teacher side.

## Assumptions

The following defaults were selected when the feature description did not specify a particular detail. Each is recorded so a reviewer can override it during `/speckit-clarify` or planning.

- **Persona anchor**: The viewer is Abdulrahman — Arabic student, Quran path level 3, family prepaid balance 750 ر.س — locked in Spec 001 §E2 and reused in Specs 002 and 003. All sample data on the four student pages anchors to this persona; both teacher pages render the persona as the *student-of-record* on the focused submission so the cross-page narrative stays coherent.
- **Featured assignment narrative**: The page-level featured assignment on `assignment-details.html` and `submission-feedback.html` is "تطبيق أحكام الميم الساكنة من سورة البقرة (الآيات 1-10)" in the "يحتاج تعديل" state with 1 attempt used and 2 remaining (grade 65/100, due 10 مايو 2026). This makes the resubmission flow the page's natural narrative without inventing an unrealistic state.
- **Featured grading narrative**: The page-level featured submission on `homework-review.html` (first or second row) and `homework-submission-details.html` is the persona's "تلاوة سورة الضحى مع تطبيق أحكام التجويد" submission — 1st attempt, submitted today, status بانتظار المراجعة. The grading form is pre-filled with the teacher's draft — grade 65, status "يحتاج تعديل"، Quran scores 85/60/75/70, mistakes count 5, three ayahs flagged — to demonstrate the page's full populated state.
- **Subject taxonomy**: Reuses the five subjects locked in Spec 003 (Quran / Arabic / English / Math / Islamic Studies). The dominant subject across the assignments index is Quran (the persona's path); ≥ 1 row of a non-Quran subject is included for variety per FR-016. The teacher pages span ≥ 3 subjects per FR-055 since the teacher reviews work across multiple courses.
- **Attempts-allowed convention**: The default attempts-allowed for any assignment is 3 (1 first submission + 2 resubmissions). This is shown on the assignment-details meta-strip and on the submission-feedback attempts-remaining caption.
- **Quran evaluation scale**: Each of the four Quran axes is scored 0-100 and rendered as a numeric value plus a horizontal bar. A score-to-letter mapping or a percentile is *not* introduced — keeping the rubric simple and consistent with the dashboard's existing percentage-based stat tiles. The axes are **diagnostic per-skill detail**, not arithmetic inputs to the final grade — the final grade in the meta-strip is a separate holistic teacher judgment (per Q1 clarification 2026-05-08). FR-044 / FR-064 carry the explanatory caption that makes this relationship explicit on both the student-facing feedback page and the teacher-facing grading form.
- **No attempts-quota visualization**: Attempts remaining is shown as a number caption ("لديك محاولتان متبقيتان من أصل 3") rather than as a fancy depleting-tokens visual. The simpler caption is consistent with the rest of the prototype's typography-first metric style.
- **Submission methods always visible**: All seven submission methods on `submit-assignment.html` are visible even when only one applies — methods that don't apply are muted with a "غير مطلوب لهذا الواجب" caption rather than removed. This preserves the "every submission method exists" narrative on every visit.
- **No real upload, no real save**: Every Submit button is `<button type="button">` with no form action; every Save button on the teacher form is the same. Inline notices on both pages call this out so reviewers don't expect persistence.
- **Resubmission state on submission-feedback**: The page renders the "needs revision with attempts remaining" state. The "rejected — no resubmission allowed" state and the "accepted — congratulations" state are documented inline as edge-case variants per the Edge Cases section.
- **Status selector default on submission-details**: The grading form's status selector defaults to "يحتاج تعديل" so the resubmission narrative is the page's primary storyline. The other four options (مقبول، مقبول مع ملاحظات، مرفوض، تأجيل المراجعة) remain visible as unselected radio cards.
- **Teacher-side → student-side status mapping (Q2 clarification 2026-05-08)**: The teacher's grading form retains all five options including the two "teacher-only" outcomes "مقبول مع ملاحظات" and "تأجيل المراجعة"; the student-facing surfaces (assignments-index pill, assignment-details meta-strip pill, submission-feedback pill) collapse these to existing student-side values — "مقبول مع ملاحظات" → "مقبول" (notes surface in the feedback panel), "تأجيل المراجعة" → "قيد المراجعة" (no separate "deferred" variant). FR-063 / FR-074 enforce this mapping on the prototype; the assignments-index sample data does NOT add new rows for these states (FR-014 row mix is unchanged).
- **Teacher persona on teacher pages**: The teacher logged in on both teacher pages is الأستاذ أحمد العمري (the same teacher who set the featured Quran assignment and graded the prior attempt) so the student-side and teacher-side narratives describe the same teacher-student pair.
- **Today's date anchor**: All "today" timestamps anchor to 2026-05-08 (the date the spec is created). All relative captions ("قبل ساعتين"، "متأخر بـ 6 أيام"، "بعد 4 أيام") compute against that date so the prototype reads as if a reviewer opened it on the day the feature was scoped.
- **Currency on grades**: Grades use the format "N / 100" (Latin digits, slash separator, "100" denominator) — no currency, no percentage symbol. This matches the dashboard's existing percentage-rendering style.
- **Audio playback stubs**: Audio file rows on both submit-assignment (uploaded preview) and submission-details (the student's submitted audio) include a play-button icon and a duration label, but no real `<audio>` element with a src — the icon is decorative and labeled "تشغيل" for screen readers. This avoids the "no real assets" risk while keeping the visual fidelity high.
- **External-link material on assignment-details**: The required external-link attachment (FR-024) points to `https://quran.com/2` — a real public Quran resource for سورة البقرة — so the no-`href="#"` rule is satisfied without inventing a fictional URL.
- **Teacher-side queue size**: The homework-review queue ships with exactly 8 rows (cap, not floor) — enough to demonstrate every status, every course, every attempt-count case without bloating the page. Pagination controls are rendered visually as a static "1 / 1 — صفحة 1 من 1" caption since 8 rows fit comfortably on one page.

## Out of scope

The following items are deliberately excluded from this feature and scheduled for later specs:

- Real file uploads, real audio recording, real video capture. The submission methods are visual stubs.
- Real grading persistence. Save buttons do not POST anywhere.
- Real notifications when a submission is graded or a resubmission is requested. The notifications page (Spec 002) already includes homework-related entries; new entries beyond that pattern are not added by this spec.
- The parent-side homework follow-up surface (`pages/parent/homework-followup.html`) — the parent dashboard already links to it, but the page itself is a future spec.
- The admin-side assignments-management surface (`pages/admin/assignments.html`) — the admin sidebar already links to it, but the page itself is a future spec.
- The teacher-side assignment-creation surface (the page where a teacher composes a new assignment with attached materials and a Quran brief) — assumed to live alongside the admin assignments page in a future spec.
- Score analytics dashboards aggregating Quran-evaluation axes across many students or many assignments. The four-axis rubric is visible per-submission on this spec; cohort-level analytics are a future surface.
- The "achievements / badges earned for homework consistency" linkage. The dashboard already shows a "بطل الواجبات" badge; tying that badge's progression to the count of accepted submissions on `assignments.html` is a future enrichment.
- Teacher-side calendar integration showing submission-due dates on a calendar grid. Out of scope; the homework-review queue is the only teacher-side surface in this spec.
- Voice-to-text annotation, AI-assisted grading suggestions, automatic Quran error detection. None of these is hinted at on the pages.
- The "rejected — no resubmission allowed" state and the "accepted — congratulations" state of `submission-feedback.html`. Documented inline as edge-case variants but not separately rendered.
- Print or PDF-export of a graded submission. No print-style sheet, no PDF download CTA.
