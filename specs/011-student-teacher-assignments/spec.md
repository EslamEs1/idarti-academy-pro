# Feature Specification: Student-Teacher Assignment & Academic Supervision Frontend

**Feature Branch**: `011-student-teacher-assignments`
**Created**: 2026-05-11
**Status**: Draft
**Input**: User description: "Build the frontend pages that manage assigning students to teachers per course, level, session type, and payment model. A student may study with multiple teachers; teacher earning differs by Arabic / Foreign student type; transfers must preserve all historical attendance / homework / report / finance links to the previous teacher; academic supervisors follow progress without necessarily teaching. Seven static HTML pages: admin's assignments list / create / details / transfer; teacher's my-assigned-students / assigned-student-details; admin's academic-supervisors. Static HTML + TailwindCSS only, no APIs, all demo data inline."

This spec (**011**) ships the **assignment-and-supervision** layer — the **operational bridge** that connects the persona (student / family from Specs 002/008), the catalog (course / level from Spec 003), the teaching staff (teachers from Specs 002/009), and the pricing/payroll backbones (rules from Spec 010, salary buckets from Spec 009, invoices from Spec 008). Every academic interaction — a homework graded by الأستاذ أحمد, a live session attended by عبد الرحمن, a settlement paid to الأستاذة منى, an invoice charged to family مؤمن — implicitly references **an underlying student-teacher-course assignment record**. Specs 001-010 referenced this record without ever surfacing it. **Spec 011 surfaces it.**

The seven new pages also unlock three previously-unaddressed operational realities in the academy:

1. **Multi-teacher students** — the persona عبد الرحمن مؤمن studies Qur'an with الأستاذ أحمد AND will, from this spec onward, also be visibly assigned to «حصص جماعية إضافية» with الأستاذة منى (mirroring his sister سارة's existing group enrollment from Spec 002). Without this spec, the platform claimed students could have multiple teachers but only showed one.
2. **Foreign-student pricing applied in context** — Spec 008 introduced the «ناطق / غير ناطق» labels at the invoice level and Spec 010 locked the pricing rules; Spec 011 is where the labels first appear **at the moment a student is bound to a teacher**, with the price preview computing live from the Spec-010 rules table. The non-Arabic-speaker anchor **عمر شودري** (a Pakistani student, family-linked to parent **محمد إقبال شودري** / family ID `FAM-2024-0089`) gets seeded here to make the foreign-tier path visibly testable.
3. **Transfer with history preservation** — when a student changes teacher mid-course, the prior teacher's attendance / homework / report / earning links MUST remain visible and untouched. This is a Constitution-VI policy rule (no silent erasure of historical state) that Specs 002-009 referenced abstractly but never demonstrated end-to-end. Spec 011 demonstrates it with **transferred-anchor assignment ASN-2024-0902** (a former student of الأستاذ أحمد, transferred 2026-03-15 to الأستاذة منى for reason «طلب ولي الأمر»; both teachers' historical records remain linked — and the row appears in الأستاذ أحمد's «طلاب سابقون» section because أحمد is the previous teacher).

The spec deliberately scopes **out**: (1) any real save / write / delete handler — Constitution I forbids backend or simulated APIs, so the create / edit / transfer / stop CTAs remain visual-only with the sanctioned-modal confirmation pattern Spec 010 introduced via `data-modal-open`; (2) any **runtime** capacity enforcement — capacity numbers shown on summary cards (e.g., "الأستاذ أحمد 12/12") are seeded data, not a calculated state from session counts; (3) any drag-drop scheduling — the preferred-days / preferred-time inputs are static form controls only; (4) editing the assignment-history timeline — historical events render as a read-only vertical timeline; (5) inviting a not-yet-existing student or teacher from inside the create-assignment form — both selects are populated by the **already-seeded** users and roster from Specs 002/008/010.

**Calendar today** = 11 مايو 2026 (carried forward from Spec 010, one day later). Anchor scenario state: the assignments-list main table seeds **18 assignment rows** (16 نشطة + 1 موقوفة + 1 مكتملة — the «إجمالي التعيينات» summary tile reads 18 with the status breakdown shown beneath it), of which **2 are «مشرف أكاديمي» supervision-only rows** (the rest are teaching rows distributed across the 4 teachers); plus **3 «بدون معلم» waiting-list rows** surfaced only in the «طلاب بدون معلم» summary tile's popover (not main-table rows); **3 transfers this month** (one for parent-request, one for level-change, one for scheduling-conflict); **2 assignments showing remaining-session counts approaching zero** (a «حصص خاصة» row 2/10 left, a «جماعية» row 1/8 left) to demonstrate the renewal-prompt state; the **1 موقوف row** demonstrates the soft-stop pattern; **6 «غير ناطق» rows** (a third of the table) demonstrate the foreign-student pricing path; **1 EGP-tier row** (الأستاذة هبة × Egyptian student) carried forward from Spec 009. The `academic-supervisors.html` page seeds its own **14 supervision rows / 4 supervisors** independently — they share the `ASN-...` identifier discipline with the main table's 2 «مشرف أكاديمي» rows but are not required to be a strict subset of the 18.

**Anchor assignment** for the deep-detail page: **`ASN-2024-1184`** — عبد الرحمن مؤمن × الأستاذ أحمد × دورة القرآن الكريم — المستوى الثالث × «معلم رئيسي» × «خاصة» × «دورة كاملة» × «ناطق بالعربية» × started 2024-09-15 × 18/24 sessions remaining × `INV-2026-0184` linked × `STL-2026-04-TCH-0042` earnings-linked. This single record is the **anchor identifier** woven across Specs 008 + 009 + 011: the invoice (Spec 008) cites it as the assignment that generated the 380-ريال subtotal; the settlement (Spec 009) cites it as one of the 22 sessions that contributed to الأستاذ أحمد's April payout; the assignment-details page (Spec 011) is the bidirectional hub showing both the invoice link and the earning-preview link in the same view — the **two-views-one-entity** dual-lens discipline applied to the assignment record itself.

## Clarifications

### Session 2026-05-11

- Q: How should `pages/admin/academic-supervisors.html` relate to the main assignments list — is it a **filtered view** of assignments where assignment-type = «مشرف أكاديمي» (sharing the same underlying entity with the assignments page), or is it a **separate set** of supervisor relationships that exist independently of teaching assignments and only appear on this page? → A: **Option A — filtered view of assignments**. The academic-supervisors page renders rows pulled from the same underlying assignment records seen on `student-teacher-assignments.html`, but filtered to only those whose assignment-type column reads «مشرف أكاديمي». Reason: this preserves the two-views-one-entity dual-lens discipline established in Spec 008 (INV-2026-0184) and Spec 010 (Finance Manager's permission matrix); a single supervisor relationship — e.g., «الأستاذة منى تشرف أكاديمياً على الطالب عمر شودري» — appears as **one row** on the main assignments page (with type-badge «مشرف أكاديمي») AND **the same row** on the academic-supervisors page (with the supervision-specific columns last-follow-up / next-follow-up / pending-parent-reports surfaced). Both views must cite the same `ASN-...` identifier verbatim. This avoids the maintenance nightmare of two parallel systems-of-record for supervision and matches the constitutional principle that one business entity has one representation.
- Q: When a student is transferred from teacher A to teacher B, should teacher A's `my-assigned-students.html` page (a) hide the student entirely (only B sees them), (b) keep them in the active list with a moved-out badge, or (c) move them to a separate "طلاب سابقون" section below the active list? → A: **Option C — separate "طلاب سابقون" section**. Teacher A's my-assigned-students page renders two stacked sections: the primary «الطلاب الحاليون» table (only currently-active assignments for teacher A) AND a secondary «طلاب سابقون — للمرجعية التاريخية» collapsible section below it (assignments whose `transferred_out_at` is non-null and whose previous-teacher was A). The historical section preserves Constitution-VI's "no silent erasure of historical state" — teacher A can still pull up old reports / homework / attendance for the transferred-out student — without confusing teacher A's day-to-day workflow by mixing past-and-present students in one table. The anchor demonstration uses **assignment `ASN-2024-0902`** (a former student of الأستاذ أحمد named «حسن المنصور», whom أحمد taught Qur'an L1 group sessions before transferring him to الأستاذة منى on 2026-03-15): the row appears in the «طلاب سابقون» section of `pages/teacher/my-assigned-students.html` (which is fixed to الأستاذ أحمد's view per FR-050 — أحمد is the previous teacher, so the section's defining rule is satisfied), AND would appear in the «الطلاب الحاليون» section of الأستاذة منى's view (the same identifier on both views — the page-pair dual-lens discipline).
- Q: For the «إيقاف التعيين» action on the assignment-details page and the «تأكيد النقل» action on the transfer page, should these use (a) a live form submit, (b) inline disabled-caption text, or (c) the sanctioned data-modal-open confirmation pattern Spec 010 introduced? → A: **Option C — sanctioned confirmation modals via `data-modal-open`**. Both consequential actions («إيقاف التعيين» on assignment-details + «تأكيد النقل» on transfer-student-teacher) trigger `<div role="dialog" class="hidden">` modals via the existing main.js handler (lines 36-44) — same pattern as Spec 010's three user-details modals and Spec 009's three settlement-action modals. Each modal carries a body caption summarizing the projected post-action state («سيتم إيقاف التعيين ابتداءً من 2026-05-12 — تبقى السجلات السابقة محفوظة لدى الأستاذ أحمد» / «سيتم نقل عبد الرحمن مؤمن من الأستاذ أحمد إلى الأستاذة منى ابتداءً من 2026-05-15 — يبقى ربط السجلات السابقة بالأستاذ أحمد دون تغيير»), a «هذا عرض مرئي فقط — لن يتم تنفيذ الإجراء فعلياً» secondary caption, and confirm / cancel buttons. **Zero new JS** — reuses the existing `data-modal-open` attribute selector. Reason: stop-assignment and transfer-teacher are exactly the consequential, family-affecting, hard-to-undo actions that Constitution V #4 authorizes confirmation modals for; they communicate the platform's expected security-and-care discipline even in a static prototype, and the modal body is the only place where the history-preservation guarantee can be reinforced at the moment of action.

At the bottom, three primary actions: «نقل المعلم» / «إيقاف التعيين» / «تعديل التعيين». سارة clicks «نقل المعلم» and lands on `pages/admin/transfer-student-teacher.html` pre-populated with the current assignment summary at top. Without Story 2, the family-trust narrative is unsupported.
