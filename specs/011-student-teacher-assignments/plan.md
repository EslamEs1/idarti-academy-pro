# Implementation Plan: Student-Teacher Assignment & Academic Supervision Frontend

**Branch**: `011-student-teacher-assignments` | **Date**: 2026-05-11 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/011-student-teacher-assignments/spec.md`

## Summary

Build seven new pages — five admin (`student-teacher-assignments`, `create-student-teacher-assignment`, `student-teacher-assignment-details`, `transfer-student-teacher`, `academic-supervisors`) and two teacher (`my-assigned-students`, `assigned-student-details`) — that ship the **assignment-and-supervision layer** of the platform: the operational bridge connecting the persona (student / family from Specs 002 / 008), the catalog (course / level from Spec 003), the teaching staff (teachers from Specs 002 / 009), and the pricing / payroll backbones (rules from Spec 010 / salary buckets from Spec 009 / invoices from Spec 008). Every academic interaction across Specs 002-010 implicitly references an underlying student-teacher-course **assignment record**; Spec 011 is the first spec to surface that record as a first-class entity with its own list, create-form, detail page, transfer flow, and dual-lens (admin + teacher) detail view.

The three clarifications resolved during `/speckit-specify` lock the spec's central decisions:

- **Q1 — Academic-supervisors page is a filtered view of the main assignments entity (clarify)**: The supervision-assignment table on `academic-supervisors.html` is derived from the same underlying Assignment records seen on `student-teacher-assignments.html` — filtered to assignment-type = «مشرف أكاديمي». A single supervisor row (e.g., «الأستاذة منى تشرف على عمر شودري») appears as **one row** on both pages, citing the same `ASN-...` identifier. This preserves the two-views-one-entity dual-lens discipline established in Spec 008 (INV-2026-0184) and Spec 010 (Finance Manager mini-matrix) and avoids the maintenance nightmare of parallel systems-of-record. Reason: one business entity has one representation; the supervision-specific columns (last-follow-up / next-follow-up / pending-parent-reports / supervision-status) are **surfaced** on the supervisors page without creating a separate underlying entity.
- **Q2 — Transferred-out students appear in a separate «طلاب سابقون» collapsible section on the prior teacher's view (clarify)**: Teacher A's `my-assigned-students.html` renders two stacked sections — the primary «الطلاب الحاليون» table (only currently-active assignments for teacher A) AND a secondary collapsible «طلاب سابقون — للمرجعية التاريخية» section (assignments whose `transferred_out_at` is non-null and whose previous-teacher was A). Reason: Constitution-VI's "no silent erasure of historical state" requires teacher A to still pull up old reports / homework / attendance for the transferred-out student, but mixing past-and-present students in one table would pollute the day-to-day workflow. The anchor demonstration uses `ASN-2024-0902` (حسن المنصور, transferred 2026-03-15 الأستاذ أحمد → الأستاذة منى — أحمد taught him Qur'an L1 group sessions before the transfer): the row appears in the «طلاب سابقون» section of `my-assigned-students.html` (which is fixed to الأستاذ أحمد's view per FR-050; أحمد being the previous teacher satisfies the section's defining rule) AND would appear in the «الطلاب الحاليون» section of الأستاذة منى's view — same identifier, two views.
- **Q3 — Stop / transfer / save-assignment consequential actions use the existing main.js `data-modal-open` sanctioned-modal pattern (clarify)**: Both consequential actions («إيقاف التعيين» on assignment-details + «تأكيد النقل» on transfer-student-teacher) trigger `<div role="dialog" class="hidden">` modals via the existing main.js handler (lines 36-44) — same pattern as Spec 010's three user-details modals and Spec 009's three settlement-action modals. Each modal carries a body caption summarizing the projected post-action state + a «هذا عرض مرئي فقط — لن يتم تنفيذ الإجراء فعلياً» secondary caption + confirm / cancel buttons. **Zero new JS** — reuses the existing `data-modal-open` attribute selector. Reason: stop-assignment and transfer-teacher are exactly the consequential, family-affecting, hard-to-undo actions Constitution V #4 authorizes confirmation modals for; the modal body is also the only place where the history-preservation guarantee can be reinforced at the moment of action.

The cross-spec data reconciliation pattern continues from Specs 002-010. The persona's locked April 2026 figures (92% attendance / 88% homework / 88-100 monthly exam / 89-100 overall / "ممتاز" / `RPT-2026-04-Q03-007`) MUST NOT be contradicted anywhere in the new pages — the persona's attendance value on the teacher's detail view is locked to «92%» verbatim, homework to «88%», and progress-summary to «ممتاز». Spec 009's anchor settlement **STL-2026-04-TCH-0042** (3,750 ريال صافي بانتظار الاعتماد) is cited on the assignment-details teacher-earning-preview tile as the destination settlement for the anchor assignment's April earnings. Spec 008's family-balance **FAM-2023-0211** (450 ريال) appears on the persona's pricing-preview / family card. The persona family (عبد الرحمن مؤمن + sister سارة + parent محمود مؤمن) and the 4 named teachers (الأستاذ أحمد / الأستاذة منى / الأستاذ خالد / الأستاذة هبة) carry over unchanged. **Three new seed personas** are introduced for this spec only:

The "no-broken-links + zero-new-JS + two-views-one-entity (at both record-level AND page-pair-level) + history-preservation-visible + cross-spec-bidirectional-anchors + Constitution-VI-pricing-at-point-of-binding" design discipline is the spec's distinguishing pattern. Eight structural levers enforce it:

- **Zero `href="#"`** — SC-002 requires a clean grep across all 7 new pages; sidebar / dropdown / footer captions use non-link text.

- **Two-views-one-entity (record-level)** — SC-004 requires the `<!-- SHARED_ASN_HEADER -->`-marked block on `student-teacher-assignment-details.html` and `assigned-student-details.html` to be byte-identical (verifiable by `diff`).
- **Two-views-one-entity (page-pair-level)** — Q1 resolution: `academic-supervisors.html` is a filtered view of the same Assignment entity rendered on `student-teacher-assignments.html`; supervision rows cite the same `ASN-...` identifiers on both pages.
- **Cross-spec bidirectional anchors** — SC-003 requires `ASN-2024-1184` to appear on ≥ 4 of the 7 new pages; FR-032 + SC-005 require `INV-2026-0184` and `STL-2026-04-TCH-0042` cited verbatim on the admin-details page; the Spec-006 figures 92% / 88% / ممتاز and the Spec-008 figure 450 ريال preserved verbatim per FR-063 + SC-005.
- **History-preservation visible** — Q2 resolution + FR-044 + SC-008: the warning-amber «إشعار حفظ السجل التاريخي» callout + the transfer-modal projected-post-action caption + the «طلاب سابقون» section on the prior teacher's view + the assignment-history timeline together demonstrate that historical state is never silently erased.
- **Sanctioned-modal pattern reuse** — Q3 resolution + FR-035 + FR-045 + SC-012: **4 distinct sanctioned-modal blocks** (save-assignment / stop-assignment / transfer / resume-assignment), each opening a dialog with the «هذا عرض مرئي فقط» caption. Each block may be triggered from multiple buttons (e.g., the per-row «إيقاف» icons on the assignments-list all point at the one shared `modal-confirm-stop-assignment` block — the established Spec-010 shared-modal pattern), so the count of `data-modal-open` *triggers* will exceed 4 while the count of distinct modal *blocks/types* is 4. All triggers reuse the existing `data-modal-open` / `data-modal-close` handler — zero new JS.
- **Constitution-VI pricing at point-of-binding** — SC-009: the create-assignment form pre-renders in the عمر شودري / «غير ناطق» / 600.00 ريال state, making the Arabic-vs-Foreign differential visible **before** the invoice is generated (Spec 008's invoice surfaced the differential after-the-fact; Spec 011 surfaces it at binding).
