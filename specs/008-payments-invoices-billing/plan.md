## Summary

Build ten new pages — two on the student side (`payment-history`, `invoice-details`), two on the parent side (`invoices`, `family-balance`), and six on the admin side (`payments`, `invoices`, `invoice-details`, `tax-settings`, `family-balances`, `create-manual-payment`) — that fulfill every constitutional sidebar promise about the academy's finance layer. A repository-wide grep before authoring confirmed that **every existing sidebar in Specs 001-007 already points to the exact file paths required by FR-001** (`payment-history.html`, `invoices.html`, `family-balance.html`, `payments.html`, `family-balances.html`, `tax-settings.html`); none of those destinations exists today, so every link 404s on every existing page that embeds those sidebars (≥ 30 student pages + 3 parent pages + 5 admin pages). Spec 008 ships them all with **zero prior-spec edits**.

The ten pages are:

- `pages/student/payment-history.html` — the persona's personal finance overview. 4 summary cards (إجمالي المدفوع 934.00 ريال — Latin digits per FR-005 / فواتير غير مدفوعة 2 / الدفعة القادمة 437.00 ريال مستحقة 2026-05-15 / رصيد العائلة المتاح 450.00 ريال) + a 6-badge status legend strip + a 7-row payment table demonstrating all 6 statuses (Paid×2 / Partially Paid / Under Review / Unpaid / Cancelled / Overdue). The "no-broken-links + multi-currency-visibility + auto-applied-balance-transparency + privacy-correct-finance-rendering" design discipline is the spec's distinguishing pattern. Six structural levers enforce it:

- **Zero prior-spec edits** — every existing sidebar already points to the new file paths (validated by repository grep before authoring); the implementation only writes new files.
- **Zero `href="#"`** — Q4 eliminated the only exception, so SC-015 now requires a clean grep across all 10 new pages.
- **Family-balance auto-application visible on every surface** — admin invoice-details timeline, parent family-balance ledger, student invoice-details payment-stream panel all carry the "تلقائي" badge on the same 67.00 ريال 2026-04-30 event.

- **VAT line on every invoice** — FR-111 mandates a separate VAT line; SC-006 grep-validates ≥ 25 invoice contexts contain "ضريبة" + "%" + currency.
- **Two-views, one-entity discipline** — INV-2026-0184 anchor invoice renders identically (same numbers, same status) on both student-side and admin-side invoice-details pages; admin adds the 3 audit-trail panels (timeline / balance-usage / admin-notes) without contradicting any student-visible figure.

Six success criteria grep-validate the discipline (SC-001 page-existence + sidebar-active; SC-004 zero broken links; SC-006 VAT-line ubiquity; SC-007 multi-currency three-surface presence; SC-009 currency-label correctness; SC-014 zero new JS lines).

## Technical Context

**Language/Version**: HTML5; CSS via TailwindCSS 3.4.x (already configured in repo); ES2020 vanilla JavaScript (zero new code added under this feature — `assets/js/main.js` remains at the Spec 001/005/006/007 baseline of 68 lines). No new dependencies introduced.
**Primary Dependencies**: Existing `tailwindcss` devDependency + `@tailwindcss/forms` + `autoprefixer` + Tajawal Google Fonts CDN — all carried over from Specs 001-007. No new libraries; no charting library (the finance surfaces use tables + cards + simple progress bars where applicable, not charts).
**Storage**: N/A — static prototype. All "data" (invoice rows, payment streams, balance transactions, family roster, tax-settings configuration, manual-payment form pre-fills) is hard-coded Arabic-first sample content embedded directly in each HTML page.
**Testing**: Manual visual review against the 15 success criteria in `spec.md` (SC-001 → SC-015) plus per-page acceptance scenarios in user stories US1-US3. Browser-based responsive validation at 375 / 768 / 1024 / 1280 px on each of the ten pages. Cross-spec reconciliation grep checks (zero prior-spec edits / zero `href="#"` / VAT-line ubiquity / persona financial figures consistency / Spec 006 academic figures untouched) run as part of the Polish phase. No automated test framework in scope.
**Target Platform**: Modern evergreen browsers (latest Chrome, Edge, Firefox, Safari) on desktop, tablet, and mobile. Pages MUST open via `file://` URLs without a dev server.
**Project Type**: Continuation of the single static frontend project at the repository root. No new top-level directories. The `pages/student/` directory grows by 2 files (`payment-history.html`, `invoice-details.html`); the `pages/parent/` directory grows by 2 (`invoices.html`, `family-balance.html`); the `pages/admin/` directory grows by 6 (`payments.html`, `invoices.html`, `invoice-details.html`, `tax-settings.html`, `family-balances.html`, `create-manual-payment.html`).
**Performance Goals**: Each page reaches first meaningful paint within ~1 s on a typical mid-range laptop opening from disk; the ten new pages add ≤ 8 KB to the compiled `output.css` after `npm run build:css`, keeping the project under the 80 KB CSS budget locked in Spec 001 (current size at the end of Spec 007 ≈ 64-65 KB → headroom ≈ 15 KB before the 80 KB ceiling). The new pages reference Tailwind utilities largely already present in the Spec 007 baseline; the JIT scan will pick up any new utility (e.g., `border-dashed` on the receipt-upload affordance, `font-tabular-nums` on numeric columns) automatically.
**Constraints**: No new JavaScript handlers — only the four sanctioned in Specs 001/006/007 carry over (sidebar drawer / dropdown / modal / tabs). The admin/invoices.html bulk-action affordance is **visually rendered but stateless** (master checkbox is a static `<input type="checkbox">` with no JS handler — toggle does not actually re-style child rows; this is the constitutional bulk-control pattern for prototype). The cascading dropdowns on `create-manual-payment.html` are **visually filtered in the rendered HTML for the demo state** (only Mu'min family + 2 children are pre-rendered as `<option>` elements when the family pre-selection is عائلة مؤمن); no JS state-change occurs when the user changes the dropdown — the spec's ALL operational guarantees stand at the rendered-HTML level. No JS-rendered invoice math; every invoice's subtotal/discount/VAT/total values are hard-coded and arithmetically verified during authoring. All Save / Pay / Apply / Add / Mark / Cancel / Download / Upload buttons are `<button type="button">` visual stubs. The Mark-Paid / Cancel-Invoice action buttons on admin/invoice-details.html visually expose their intent (success-tone / danger-tone styling) without any state mutation.
**Scale/Scope**: 10 pages (2 student + 2 parent + 6 admin); 0 new sidebar entries on any role (every entry referenced is constitutionally enumerated and already present in prior-spec sidebars); 0 header changes; 7 entity types modeled in static HTML across the 10 pages; ~250 unique sample-data records (7-row persona payment history with full financial breakdown × 7 invoices on `student/payment-history`; 1 anchor invoice × ~20 attributes on student `invoice-details`; ≥ 8 parent-invoice rows × 8 attributes on `parent/invoices`; 7-row family-balance ledger × 6 attributes + 4 sub-summary tiles + hero card on `parent/family-balance`; ≥ 10-row admin payments table × 10 attributes + 5 summary cards on `admin/payments`; ≥ 12-row admin invoices table × 9 attributes + 5 summary cards + 6 filter controls on `admin/invoices`; same anchor invoice + 3 admin-only panels (timeline 3 events / balance-usage / admin notes 2) + 5-action bar on `admin/invoice-details`; full tax-settings configuration with two example-invoice cards on `admin/tax-settings`; 6-row families table + 3 summary tiles + filter controls on `admin/family-balances`; full manual-payment form with 7-option method dropdown + cascading filters + receipt-upload affordance on `admin/create-manual-payment`).
