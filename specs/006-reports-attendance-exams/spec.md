# Feature Specification: Reports, Attendance, and Exams Frontend

**Feature Branch**: `006-reports-attendance-exams`
**Created**: 2026-05-08
**Status**: Draft
**Input**: User description: "Build reporting pages that make parents confident and help students understand their progress. Reports must cover attendance, homework, exams, teacher notes, monthly progress, strengths, weaknesses, and next-month recommendations."

## Overview

This feature ships the **Reports, Attendance, and Exams** layer that every prior spec has been pointing at. The student dashboard, weekly-plan, and learning-journey sidebars already link to `reports.html` and `exams.html`; the parent dashboard links to `my-children.html`, `child-reports.html`, and `attendance.html`; the teacher dashboard links to `student-reports.html` and `attendance.html`. None of these destinations exist today — every link 404s. Spec 006 closes that gap with eight pages — five on the student side, two on the parent side, one on the teacher side — that turn the academy's promise of *we measure, we report, we recommend* into a visible, parent-confident surface.

The spec adds:

1. `pages/student/reports.html` — the multi-month performance hub: month selector, five summary cards (active classes / attendance / homework completion / quiz average / overall progress), and a per-course detail table (course / teacher / attendance / homework / quiz avg / status / "Details" CTA → `monthly-report.html`).
2. `pages/student/monthly-report.html` — a single month's deep-dive report card: student info, month, overall evaluation, attendance + homework + exams summaries, teacher notes, strength points, improvement points, recommended plan for next month, parent-friendly summary, and a visual "تنزيل PDF" button.
3. `pages/student/attendance.html` — the full attendance log: summary cards (total sessions / present / absent / late / excused / attendance rate), a session-by-session table (date / course / session / teacher / status / notes), four named status badges (حاضر / غائب / متأخر / معذور), and an attendance-trend visual (sparkline / bar across months).
4. `pages/student/exams.html` — the comprehensive exams hub: filters (course / month / status), three sectioned tabs ("اختبارات قادمة" / "اختبارات مكتملة" / "اختبارات شهرية"), and exam cards/table rows (title / course / date / score / status / CTA → `exam-details.html`).
5. `pages/student/exam-details.html` — a single exam's deep-dive: exam summary (course, teacher, date, duration), score + percentage + status, question-categories breakdown (e.g., حفظ / تجويد / فهم), strengths, weaknesses, teacher recommendations, and — when applicable — a related-certificate panel with a CTA to `certificate-preview.html`.
6. `pages/parent/my-children.html` — the parent's family overview: cards for each child (name, current level, attendance %, homework %, last report date, payment status, "View report" CTA → `child-reports.html`). _Filename matches the existing parent-dashboard sidebar + body link to `my-children.html` — zero prior-spec edits needed._
7. `pages/parent/child-reports.html` — the parent-focused report: clear-language attendance / homework / exams / teacher notes / behavior-engagement panels, a tri-state progress direction indicator (متحسّن / مستقر / يحتاج إلى دعم), suggested parent action, a "تنزيل التقرير" button, and a "مشاركة عبر واتساب" visual button. _Filename matches the existing parent-dashboard sidebar + body link to `child-reports.html`._
8. `pages/teacher/student-reports.html` — the teacher's authoring surface for a single student's report: student information, teacher evaluation sections (attendance / homework / exam performance), notes, recommendation form UI, and a "حفظ التقرير" visual button. _Filename matches the existing teacher-dashboard sidebar + body link to `student-reports.html`._

**Out of scope for this spec** — parent-role and teacher-role attendance variants. The parent dashboard's `attendance.html` link (line 63 sidebar + line 364 body) and the teacher dashboard's `attendance.html` link (line 79 sidebar + line 374 body) are inherited prior-spec broken links pointing to `pages/parent/attendance.html` and `pages/teacher/attendance.html` respectively. The user's brief specifies a single attendance page (`pages/student/attendance.html`) and Spec 006 honors that scope strictly. The two parent / teacher attendance shell variants are documented in Assumptions as a known follow-up — to be addressed in a later spec or as a small post-006 fix-up that mirrors the student attendance page into the parent and teacher chromes.

The feature is purely visual / frontend. No real PDF download, no real WhatsApp share, no real form persistence. The point is to make three propositions unmistakable on the screen:

1. **The academy measures every dimension** — attendance, homework, quizzes, monthly exams, behavior, engagement — and surfaces them in one place per audience (student / parent / teacher), each with the right reading level for that audience.
2. **Parents can act on a report at a glance** — the parent's `child-reports` distills clinical metrics into clear-language summaries, a tri-state progress indicator (متحسّن / مستقر / يحتاج إلى دعم), and a single suggested parent action, so a working parent can read and decide in under a minute.
3. **Teachers author the truth** — the teacher's `student-reports` is the upstream form whose output is what students and parents read; rendering its evaluation sections + recommendation form + Save button shows the academy's "teachers actively evaluate, not just teach" promise.

The cross-page narrative anchors to the **persona family** locked since Spec 002:
- **Primary student**: عبد الرحمن مؤمن (ع.م) — الصف السادس — Quran path Level 3 in progress at 60%, 4 certificates earned, 7 badges, 5 monthly exams passed (cumulative average 86.6%).
- **Sibling**: سارة مؤمن (س.م) — الصف الثالث — Arabic path Level 1, "يحتاج إلى دعم" status (already pinned on the parent dashboard's children card).
- **Parent**: ولي أمر الطالب عبد الرحمن (already labeled on the parent dashboard).
- **Primary teacher**: الأستاذ أحمد بن عبد الله (أ.أ) — Quran instructor (locked since Spec 005).
- **Secondary teacher**: الأستاذة منى سعد (م.س) — Arabic instructor for sibling سارة and for the persona's Arabic-language elective (introduced in this spec to complete the multi-course report).

Every count, every percentage, every grade reconciles with what the dashboard / weekly-plan / learning-journey / my-courses / monthly-exams-passed already claim.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Opening the Student Reports Hub (Priority: P1) 🎯 MVP

A student opens the **Reports** page from the sidebar (or from the dashboard's "عرض التقارير" link, the weekly-plan footer, or the learning-journey footer). They want a multi-month performance hub: which month am I looking at, how am I doing across all my active courses on the five academy-watched dimensions (attendance / homework / quiz average / overall progress / active class count), and per-course where can I drill in for the full monthly report.

**Why this priority**: This is the canonical reports surface — the entry point every student-side prior-spec page already targets. Without it, the dashboard / weekly-plan / learning-journey reports links 404. P1 because it's also the *student's* answer to "how am I doing this term" — the inverse of the parent-side `child-reports`.

**Independent Test**: Open `pages/student/reports.html` from the file explorer. Without any other context, the viewer must be able to answer in 30 seconds: "Which month am I viewing? How many active classes does this student have? What's their attendance rate? Their homework completion rate? Their quiz average? Their overall progress score? Per course, who teaches it, how is the student doing on attendance/homework/quizzes, and where do I open the full monthly report?" — entirely from the page.

**Acceptance Scenarios**:

1. **Given** a student opens reports, **When** they look at the top, **Then** they see h1 "تقاريري الشهرية" + a one-line subtitle ("تابع تقدّمك عبر الأشهر — اختر الشهر الذي تريد مراجعته") and the persona greeting in the header chrome.
2. **Given** the student wants to switch months, **When** they look near the title, **Then** they see a month-selector control rendered as a horizontal chip strip showing the last 6 months (ديسمبر 2025 / يناير 2026 / فبراير / مارس / أبريل / مايو 2026 — current) with the selected month visually highlighted (bg-accent-100, ring-2 ring-accent-500). The strip is a static visual control (the active selection drives what the rest of the page shows; switching months is documented as a future-spec interaction).
3. **Given** the student wants a workload glance for the selected month, **When** they look below the selector, **Then** they see exactly five summary cards (1 col mobile / 2-3 col tablet / 5 col desktop): "الفصول النشطة" (3), "الحضور" (92%), "إنجاز الواجبات" (88%), "متوسط الاختبارات" (86.6%), "التقدّم الإجمالي" (89%) — each with distinct accent color paired with an icon, never color alone.
4. **Given** the student wants the per-course detail, **When** they look at the table, **Then** they see ≥ 3 course rows (the student's active classes in May 2026): القرآن الكريم — المستوى الثالث (الأستاذ أحمد بن عبد الله), اللغة العربية — المستوى الرابع (الأستاذة منى سعد), الدراسات الإسلامية — المستوى الثاني (الأستاذ خالد العبدلي). Each row shows columns: الدورة, المعلم, الحضور (%), الواجبات (%), متوسط الاختبارات (%), الحالة (status pill: ممتاز / جيد جداً / جيد / يحتاج إلى دعم), and a "عرض التفاصيل" CTA → `monthly-report.html`.
5. **Given** the page is responsive, **When** the student opens at mobile width (< 768 px), **Then** the per-course table collapses to stacked single-column cards preserving every cell, and the month-selector becomes horizontally scrollable rather than wrapping.
6. **Given** the student wants the longer-form view, **When** they look at the page footer, **Then** they see a primary "عرض التقرير الشهري الكامل" CTA → `monthly-report.html` (defaults to the selected month).

### User Story 2 — Reading the Full Monthly Report (Priority: P1)

A student taps "عرض التقرير الشهري الكامل" on the reports hub (or "عرض التفاصيل" on any per-course row, or "عرض التقرير الكامل" from the dashboard's monthly-report card preview) and lands on the **Monthly Report** page. They want a printable, parent-shareable, single-page report card for one month: who they are, which month, the academy's overall evaluation of them, and the four narrative sections (attendance / homework / exams / teacher notes) plus the three reflection panels (strength points / improvement points / next-month plan), capped with a parent-friendly summary paragraph and a "Download PDF" visual button.

**Why this priority**: This is the document the parent and the student will actually read together at the end of the month. It is the single most important "trust the academy" surface per the user's brief — the canonical artifact that every other report page either previews or summarizes. P1 because without it the brief's "make parents confident" promise has no tangible deliverable.

**Independent Test**: Open `pages/student/monthly-report.html` from the file explorer. A viewer must be able to answer: "Whose report is this? Which month? What is the academy's overall evaluation? How was attendance / homework / exams summarized? What did teachers say in their notes? What are the student's strength points and improvement points? What's the recommended plan for next month? What's the parent-friendly summary? Where is the Download PDF button?" — entirely from the page.

**Acceptance Scenarios**:

1. **Given** the student opens monthly-report, **When** they look at the top, **Then** they see a report-card-styled header panel (cream/parchment background, soft border, deep-navy text in display weight) containing: academy logo + "أكاديمية إدارتي" wordmark, "التقرير الشهري" title, the month label "أبريل 2026", and a student-info strip (avatar, name "عبد الرحمن مؤمن", level "الصف السادس — مسار القرآن الكريم", report ID "RPT-2026-04-Q03-007"). The header is the "official report" affordance — visually distinct from the rest of the chrome.
2. **Given** the student wants the academy's verdict, **When** they look below the header, **Then** they see an "التقييم العام للشهر" panel with a large status pill ("ممتاز" — success-100 with star icon), a horizontal score bar showing "89 / 100", and a 1-2-sentence overall evaluation quote attributed to "إدارة الأكاديمية".
3. **Given** the student wants the four data summaries, **When** they look at the next row, **Then** they see exactly four mini-summary cards (1 col mobile / 2 col tablet / 4 col desktop): "ملخص الحضور" (92% — 11/12 sessions, with a 1-line caption "حضور ممتاز هذا الشهر"), "ملخص الواجبات" (88% — 7/8 assignments accepted, "تسليمات منتظمة في موعدها"), "ملخص الاختبارات" (88/100 — اختبار أبريل الشهري, "أداء قوي في الحفظ"), "ملاحظات المعلمين" (3 ملاحظات — "تواصل بنّاء من الأساتذة"). Each card has its own accent + icon and links to a more-detailed deeper page (الحضور → `attendance.html`, الواجبات → `assignments.html`, الاختبارات → `exam-details.html`, ملاحظات → in-page anchor).
4. **Given** the student wants the teachers' voices, **When** they look at the "ملاحظات المعلمين" section, **Then** they see ≥ 3 teacher-note quote blocks each with: teacher avatar + name, course chip, the date the note was authored, and a 2-4 sentence Arabic feedback message. Sample notes: (a) الأستاذ أحمد بن عبد الله — القرآن الكريم: "أداء عبد الرحمن في الحفظ هذا الشهر ممتاز، وقد أتقن أحكام الميم الساكنة. يحتاج إلى تركيز إضافي على التجويد في الآيات الطويلة." (b) الأستاذة منى سعد — اللغة العربية: "تحسن واضح في القراءة الجهرية، مع التركيز على علامات الترقيم في الكتابة الإنشائية القادمة." (c) الأستاذ خالد العبدلي — الدراسات الإسلامية: "حضور والتزام مثاليّان، ومشاركة فاعلة في النقاشات الصفّية."
5. **Given** the student wants the reflection panels, **When** they look at the next section, **Then** they see three side-by-side panels (stacked on mobile): "نقاط القوّة" (success-tinted) with ≥ 4 bullets (الحفظ المتقن لسورة الفجر, الالتزام الكامل بمواعيد الحصص, جودة الكتابة الإنشائية, الفهم العميق للسيرة النبوية); "نقاط للتحسين" (warning-tinted) with ≥ 3 bullets (تطبيق أحكام التجويد في الآيات الطويلة, تنويع المفردات في التعبير الكتابي, زيادة سرعة القراءة الجهرية); "خطة الشهر القادم — مايو 2026" (info-tinted) with ≥ 4 bullets (إكمال المستوى الثالث من القرآن وبدء التحضير للمستوى الرابع, مراجعة قواعد الإملاء الأسبوعية, جلسة فردية مع الأستاذ أحمد لتجويد الآيات الطويلة, اختبار شهر مايو في 28 مايو 2026).
6. **Given** the report needs to land for a parent, **When** the student looks at the bottom, **Then** they see a "ملخص لولي الأمر" panel with a 4-6-sentence parent-readable paragraph in clear, non-technical Arabic (e.g., "عبد الرحمن قضى شهراً ممتازاً في الأكاديمية. حافظ على حضور كامل تقريباً، وأنجز معظم واجباته بجودة عالية، وحصل على 88% في الاختبار الشهري لشهر أبريل. أساتذته راضون عن مستواه وأشاروا إلى نقاط محدّدة للتطوير في شهر مايو. يمكنكم متابعة هذه النقاط معه في البيت لتعزيز تقدّمه.").
7. **Given** the report is meant to be archivable, **When** the student looks at the action cluster below the parent-summary, **Then** they see exactly three buttons: primary "تنزيل PDF" (`<button type="button">` — visual stub), secondary "طباعة" (visual stub), tertiary "مشاركة مع ولي الأمر" (visual stub). A small inline notice clarifies "هذا نموذج تجريبي — لا يتم تنزيل ملف فعلي عند النقر." (matches the prototype-notice convention from Spec 003/004/005).
8. **Given** the page is responsive, **When** the student opens at mobile width, **Then** the four-card row collapses to single-column, the three reflection panels stack vertically, and the parent-summary panel's typography stays readable (≥ 16 px line-height ≥ 1.7).

### User Story 3 — Parent Opens the Children Overview (Priority: P1)

A parent opens **My Children** from the parent sidebar (or from the parent-dashboard's "عرض تفاصيل الأبناء" CTA). **Why this priority**: This is the parent's primary navigation surface — it answers "who do I focus on this week?" before they drill into a single child. The parent dashboard already pins two children (عبد الرحمن مؤمن "ممتاز" + سارة مؤمن "يحتاج إلى دعم") and links here. P1 because it's the parent's daily entry point and the only multi-child surface in the spec.

**Independent Test**: Open the parent children-overview page from the file explorer. A viewer must be able to answer: "How many children does this family have at the academy? What is each child's name, current level, attendance %, homework %, last-report date, and payment status? Which child needs the most attention this week? Where do I open each child's full report?" — entirely from the page.

**Acceptance Scenarios**:

1. **Given** the parent opens the children-overview page, **When** they look at the top, **Then** they see h1 "أبنائي" + a one-line subtitle ("متابعة شاملة لتقدّم كل طفل — اضغط على أيّ طفل لقراءة تقريره الكامل") and the parent persona greeting in the header chrome ("ولي أمر الطالب عبد الرحمن").
2. **Given** the parent wants the family at a glance, **When** they look at the cards, **Then** they see exactly 2 child cards in a responsive grid (1 col mobile / 2 col tablet+): (Card 1) عبد الرحمن مؤمن — الصف السادس — مسار القرآن الكريم — حضور 92% — واجبات 88% — آخر تقرير 30 أبريل 2026 — السداد: مدفوع — status pill "ممتاز" (success-100 + star icon) — primary CTA "عرض التقرير الكامل" → child-report page; (Card 2) سارة مؤمن — الصف الثالث — مسار اللغة العربية — حضور 78% — واجبات 65% — آخر تقرير 30 أبريل 2026 — السداد: مدفوع — status pill "يحتاج إلى دعم" (warning-100 + alert icon) — primary CTA "عرض التقرير الكامل" → child-report page (with sibling-switch context).
3. **Given** the parent wants more signal per child, **When** they look at any card, **Then** they also see secondary chips below the primary CTA: "الفواتير" (→ `invoices.html`), "الحضور" (→ `attendance.html`), "الواجبات" (→ `homework-followup.html` — placeholder anchor), and a small avatar + level icon strip.
4. **Given** the parent has a quick action, **When** they look near the page header, **Then** they see a small "أضف طفلاً جديداً" / "اطلب إضافة طفل" CTA (visual stub) — a placeholder for the future "request to enroll another child" flow.
5. **Given** the page is responsive, **When** the parent opens at mobile width, **Then** the cards stack to 1 column, the secondary chip row wraps, and the page is readable without horizontal scroll.

### User Story 4 — Parent Reads the Child Report (Priority: P1)

A parent taps "عرض التقرير الكامل" on a children-overview card (or "قراءة التقرير الكامل" on the parent dashboard's monthly-report card, or follows a "Reports" entry from the parent sidebar) and lands on the **Child Report** page. They want a parent-focused, plain-language report of their child: clear-language attendance / homework / exams summaries, teacher notes, behavior + engagement signals, a tri-state progress direction (متحسّن / مستقر / يحتاج إلى دعم), a single suggested parent action, a "Download report" button, and a "Share via WhatsApp" visual button.

**Why this priority**: This is the literal embodiment of the brief's "make parents confident" objective. It is the page the parent will read on their phone after the school day, decide if the child needs help, and (visually) share with extended family. Without this page, the brief's primary deliverable is missing. P1, highest priority alongside Story 2.

**Independent Test**: Open the parent child-report page from the file explorer. A viewer must be able to answer: "Whose report is this? What's the academy's progress direction for this child (improving / stable / needs support)? How is attendance, homework, exams summarized in plain language? What did the teachers say? How is the child's behavior and engagement? What single action does the academy suggest the parent take? Where do I download the report or share it via WhatsApp?" — entirely from the page.

**Acceptance Scenarios**:

1. **Given** the parent opens the child-report page, **When** they look at the top, **Then** they see a report-card-styled header showing: "تقرير عبد الرحمن — أبريل 2026" (h1), child avatar + name + level chip, parent name, the month, and a sibling-switcher chip strip (عبد الرحمن — selected / سارة) so the parent can flip to the other child's report from the same surface (visual control — clicking سارة conceptually navigates to a sibling-context variant of this page; for the prototype, both states are documented but only the عبد الرحمن variant is rendered, with the سارة chip linking to `?child=sara` placeholder).
2. **Given** the parent wants the headline verdict, **When** they look below the header, **Then** they see a single large "اتجاه التقدّم" panel with a tri-state indicator: a 3-step horizontal track labeled "يحتاج إلى دعم — مستقر — متحسّن" with the "متحسّن" step highlighted (success-700 fill + check icon), the other two steps in muted slate-100. Below the track: a 1-2-sentence parent-readable rationale ("عبد الرحمن أظهر تحسناً ملحوظاً في الحفظ والالتزام هذا الشهر مقارنة بمارس.").

### Edge Cases

The following states are documented as edge-case patterns. Where the prototype renders one variant, the alternative is documented inline (annotation or comment) so the design intent is preserved without doubling the page count.

- **Month with no data (empty reports)**: Reports hub MUST document an empty state in a comment block ("لا توجد بيانات لهذا الشهر — اختر شهراً آخر، أو ابدأ تسجيل حصصك."). Prototype renders the populated أبريل 2026 month.
- **First-month student (no prior reports)**: Documented on `monthly-report.html` as a comment annotation — when this is the first report, the previous-months reference degrades to a "هذا أول تقرير شهري لك — مرحباً برحلتك في إدارتي!" empty-state.
- **Child report with "يحتاج إلى دعم" tri-state**: The prototype renders the متحسّن variant for عبد الرحمن. The "يحتاج إلى دعم" variant is documented in a comment for سارة (the sibling). The tri-state component MUST be rendered as a 3-step horizontal track regardless of which step is active.
- **Single-child family**: The children-overview page MUST document a single-child empty state in a comment block ("عائلتك تضمّ طفلاً واحداً مسجّلاً — يمكنك طلب إضافة طفل جديد في أيّ وقت."). Prototype renders the 2-child variant matching the parent dashboard's existing pin.
- **Parent with no children enrolled yet (unusual)**: Documented as a comment empty state ("لم يتم تسجيل أيّ طفل بعد — اطلب التسجيل من الإدارة لبدء المتابعة.").
- **Teacher-side prefilled-from-existing vs blank-new report**: The prototype renders the prefilled variant (April 2026 report on عبد الرحمن). The blank-new variant is documented in a comment annotation.
- **Exam not yet graded**: Documented on `exam-details.html` as a comment — when the exam is "بانتظار التصحيح", the score panel renders "—" with status pill "قيد المراجعة" and the category breakdown / strengths / weaknesses panels are hidden behind a "نتيجة الاختبار لم تظهر بعد" placeholder.
- **Student failed an exam**: Documented on `exam-details.html` and `exams.html` as a comment — the "يحتاج إعادة" variant is shown on at least one row of the completed-exams tab; the corresponding details page renders the alternative result-panel (warning-100 background + retake-CTA "موعد الإعادة قريباً سيظهر هنا").
- **Teacher edits an already-saved report**: Documented as a comment — the "هذا التقرير محفوظ — أيّ تعديل يستلزم إعادة الحفظ" caption appears at the top of the form.
- **Print view of monthly-report / child-report**: Both pages should be print-friendly via a `@media print` block (extending the one added in Spec 005's `assets/css/input.css`) that hides the sidebar, header, action cluster, and prototype-notice — leaving only the report card visible.
- **Long teacher-note overflow**: Notes must wrap gracefully without breaking the report card layout. Documented as a CSS responsiveness concern.
- **All-attendance-100% student**: Documented as a comment on `attendance.html` — the empty "غائب / متأخر" status filter chips must still render the chip UI even when zero rows match.

## Requirements *(mandatory)*

### Functional Requirements

#### Cross-cutting (apply to every page in this spec)

- **FR-001**: Every page MUST render with `lang="ar"`, `dir="rtl"`, the project's TailwindCSS output, the existing Tajawal Google Fonts CDN reference, and the existing role-sidebar + header chrome from the corresponding role's prior pages (Student / Parent / Teacher).
- **FR-002**: No new JavaScript MUST be introduced beyond the four sanctioned handlers in `assets/js/main.js` (sidebar drawer / dropdown / modal / tabs). Where this spec uses tabs (e.g., `exams.html` Story 7), it MUST use the existing `data-tab`/`data-tabpanel` handler from Spec 002/003/004 with no new JS.
- **FR-003**: No real download / share / save / print / WhatsApp action MUST execute. Every Save / Download / Print / Share button is a `<button type="button">` visual stub. Every page that exposes such buttons MUST include the standard prototype-notice ("هذا نموذج تجريبي — لا يتم تنزيل ملف فعلي عند النقر." or its action-appropriate variant).
- **FR-004**: Every status badge across every page MUST follow the constitution v1.0.1 vocabulary AND the same color + icon + text triple defined in Spec 001/004/005 (color is never the sole indicator). The four named attendance statuses (حاضر / غائب / متأخر / معذور) MUST use a distinct color hue per status: success / danger / warning / info respectively, paired with check / x / clock / shield icons.
- **FR-005**: Every cross-page reference (course names, teacher names, student name, sibling name, parent name, dates, percentages, scores, certificate IDs) MUST reconcile across this spec's eight pages AND MUST reconcile with prior-spec anchors: عبد الرحمن مؤمن (persona), سارة مؤمن (sibling), الأستاذ أحمد بن عبد الله (Quran), الأستاذة منى سعد (Arabic — newly introduced here), 4 certificates earned, 7 badges, 5 monthly exams passed (cumulative average 86.6%), 2 levels completed, family balance 750 ر.س, current month أبريل 2026 (just-completed) and مايو 2026 (in-progress).
- **FR-006**: All numeric data (scores, percentages, attendance counts, dates) MUST use Latin (Western Arabic) digits inside Arabic copy, matching the convention enforced since Spec 001. Eastern Arabic-Indic digits (٠-٩) are forbidden.
- **FR-007**: No `href="#"` placeholder links. Internal links MUST point to a real existing HTML file in the project, OR `href="#planned"` for future-spec destinations (the same convention used in Spec 005).
- **FR-008**: Every form input on `student-reports` (Story 5) MUST have a visible `<label>` with a matching `for` attribute paired to the input's `id` (constitution + Spec 005 FR-079 parity).
- **FR-009**: All eight pages MUST be responsive at 375 / 768 / 1024 / 1280 px. Tables MUST collapse to single-column cards at < 768 px without horizontal page scroll (constitution gate 2).
- **FR-010**: All "Download" / "Print" / "Share" / "Save" / "WhatsApp" actions MUST appear ONLY on pages where the user's brief explicitly calls for them (`monthly-report.html`, the child-report page, the student-report page). They MUST NOT appear on `reports.html`, `attendance.html`, `exams.html`, `exam-details.html`, or the children-overview page. (Mirrors Spec 005's "download-only-on-preview" design discipline.)
