This spec (**010**) ships the **administrative-control layer** — the back-of-house surfaces that govern who can do what across the entire platform: roles, permissions, users, platform settings, pricing rules, student-type rules, notifications, and the academy-wide audit trail. It closes the last two unlinked entries on the constitutional Admin sidebar — **«الأدوار والصلاحيات»** (`roles-permissions.html`) and **«إعدادات المنصة»** (`platform-settings.html`) — both of which are already referenced from every prior-spec admin sidebar (verified `grep`: 18 refs to `roles-permissions.html` + 34 refs to `platform-settings.html` across `pages/admin/*.html`). The other eight new files (`role-details`, `create-role`, `users`, `user-details`, `pricing-rules`, `student-type-settings`, `notification-settings`, `audit-log`) are reachable from those two entry points and from each other; **zero prior-spec edits** are required.

The spec deliberately scopes **out**: (1) any real authentication or session management — Constitution I forbids backend or simulated APIs, so login/logout flows remain cosmetic; (2) any actual permission enforcement — the matrix is a **visual representation** of the academy's policy, not a runtime gate; (3) inviting external users via email — the user-creation form is reserved for a future spec (this spec ships only the visual list + detail + read-mostly experience for existing seeded users); (4) granular per-record permissions (e.g., "Teacher A can only see Student X") — module-level + action-level granularity is the ceiling.

**Calendar today** = 10 مايو 2026 (carried forward from Spec 009). Anchor admin scenario state: the academy operates 7 system roles, 14 named seeded users (2 Admin / 3 Finance Manager / 5 Teacher seeds — including the persona-anchor teacher الأستاذ أحمد + الأستاذة منى + الأستاذة فاطمة + الأستاذ يوسف + الأستاذة هبة — / 1 Parent / 1 Student / 1 Support / 1 Operations), the platform is configured for "أكاديمية إدارتي للتعليم" (KSA primary, EGP secondary), 8 pricing rules are active across course + session + foreign-tier + Egypt-tier, the 2 student types (ناطق / غير ناطق) are active and locked, all 6 notification categories are enabled with the 4 channels (Platform always-on, Email mostly-on, WhatsApp selectively-on, SMS payment-only), and the audit-log demonstrates 24 chronological events spanning 11 modules over the last 7 days (2026-05-04 → 2026-05-10).

**Anchor admin user** for the dual-lens demo: **سارة محمد عبدالله** (`USR-2024-0007`) — موظفة المالية / Finance Manager — the same SARA.M who created the manual adjustment on STL-2026-04-TCH-0042 in Spec 009 and the manual payment events in Spec 008. Her **role-details** view + **user-details** view together demonstrate the partial-permission pattern (full Finance / Invoices / Tax / Family-Balance / Teacher-Finance / Teacher-Advances access + read-only Reports / Students / Teachers + zero Roles & Permissions / Platform Settings / Audit Log access — i.e., she sees everything financial but cannot edit who-can-do-what nor change platform-wide configuration). ## Clarifications

### Session 2026-05-10

- Q: Pricing-rules-form retroactivity discipline — should `pages/admin/pricing-rules.html` carry the same "تسري إلى الأمام فقط" callout that Spec 009 introduced for the salary-rules form (Q3), or should pricing rules be allowed to retroactively recalculate already-issued invoices? → A: Option A — **mirror Spec 009's Q3 pattern**. A permanent regulatory callout «تسري القاعدة على الفواتير الجديدة فقط — لا تُعدَّل الفواتير المستحقة سابقاً» renders beside the rule-form's effective-date picker, plus a small footnote `* جميع قواعد التسعير تُطبَّق إلى الأمام من تاريخ السريان.` below the rules table. Reason: financial transparency for invoice retroactivity has the same family-trust and tax-audit implications as salary retroactivity for teachers; treating pricing rules as forward-only matches Saudi VAT-compliance norms and avoids any "the academy charged me a different rate last week than this week for the same course" complaint pattern. The Spec-008 anchor invoice INV-2026-0184 (April, 380 ريال subtotal) is therefore not affected by any rate change shown on this page — and the page documents that explicitly.
- Q: Audit-log export discipline — should `pages/admin/audit-log.html` expose a "تصدير" button to satisfy the "Export" permission action listed in the role-details matrix, and if so should it be functional or visual-only? → A: Option B — **render a visual-only «تصدير CSV» button in the page header next to the date-range filter**, styled as a secondary outline button with a download icon, carrying a Constitution-V-compliant title attribute `title="عرض مرئي فقط — لا يتم تنزيل ملف فعلي"` and no `download` attribute. Reason: the Export action is one of the seven permission verbs in the user input's role-details matrix; surfacing it on the audit-log page (the most export-heavy admin surface) is the single best place to make the Export permission visually concrete. A functional CSV download would require either a bundled static file (Constitution II permits this only when the file is real and meaningful) or a JS handler (Constitution V forbids new JS for this purpose) — the visual-only button with the explanatory tooltip is the cleanest reconciliation.
- Q: User-details account-disable affordance — when an admin views another user's profile on `pages/admin/user-details.html`, should the "تعطيل الحساب" / "إعادة تعيين كلمة المرور" / "إنهاء الجلسات" actions render as live CTAs (would need handlers + sanctioned modals like Spec 009 Q1), as inline disabled-caption text (zero-href-hash discipline from Spec 008/009), or as a sanctioned confirmation-modal trigger group (Constitution V #4)? → A: Option C — **render the three actions as a sanctioned-modal trigger group**. The three buttons («تعطيل الحساب» / «إعادة تعيين كلمة المرور» / «إنهاء جميع الجلسات») use the existing `data-modal-open` attribute pattern wired in main.js since Spec 002 (lines 36-44) — same pattern as Spec 009 Q1's settlement-action modals — opening three distinct `<div role="dialog" class="hidden">` modals with confirm/cancel buttons and a "هذا عرض مرئي فقط" secondary caption. Reason: account-disable / password-reset / session-termination are exactly the kind of consequential actions that deserve a confirmation-modal pattern under Constitution V #4 (sanctioned modal use); the inline-disabled-caption pattern (Spec 008/009) was reserved for **non-action** captions like "تعديل بيانات الحساب" — these are real admin actions even in a static prototype, and visualizing the confirmation step is essential to communicate the security-discipline expected of the platform. **Zero new JS** — reuses the same data-attribute selectors already in main.js.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Super-admin governs the academy's roles, permissions, and people (Priority: P1)

The super-admin **محمد بن عبدالعزيز الإدريسي** (المدير التنفيذي) opens the admin sidebar and lands on **«الأدوار والصلاحيات»** to do his Sunday governance review. He scans the seven role cards (Admin / Finance Manager / Teacher / Parent / Student / Support / Operations) — each showing a user count, a permissions count, an active/inactive status, and a «عرض/تعديل» CTA — then clicks into «Finance Manager» to inspect what سارة and her two finance colleagues are actually authorized to do. The role-details page renders the full **19-module × 7-action permission matrix** (Dashboard / Students / Families / Teachers / Courses / Live Sessions / Assignments / Exams / Certificates / Reports / Payments / Invoices / Tax Settings / Family Balance / Teacher Finance / Teacher Advances / Social Hub / Platform Settings / Audit Log × View / Create / Edit / Delete / Approve / Export / Financial access) with check / dot / dash icons making the partial-permission pattern instantly readable. Satisfied, he navigates to **«المستخدمون»** to scan the 14-user table, applies the role filter to «Finance Manager», and clicks سارة's row to open her user-details page where he can see her assigned role, her permissions summary (mirrored from the role she's assigned), her account status (نشط), her last-login info (today 09:12), and a 5-event activity preview pulled from the audit log. From the user-details page he can also use the three sanctioned-modal trigger buttons (تعطيل الحساب / إعادة تعيين كلمة المرور / إنهاء جميع الجلسات) — each opens a confirmation modal with a "عرض مرئي فقط" caption, demonstrating the security-discipline of the platform without performing any real action.

**Why this priority**: This is the single governance slice that defines who-does-what across the entire academy. Without it, every other admin page is operating in an unprincipled vacuum: there's no visible account of why سارة can adjust a teacher's settlement but the receptionist أحمد can't, why parents see their own children's invoices but not other families', or why teachers cannot view the platform's tax settings. Closing this slice makes every prior-spec admin restriction story concrete and demoable to the academy's owner. It also unblocks Constitution Quality Gate 5 ("every navigation link points to a real existing HTML file") on the last two unlinked entries of the Admin sidebar.

Finally open `pages/admin/user-details.html` (the file is static and hard-coded for سارة) and verify the profile card, role assignment card, permissions summary mini-matrix, account status card, last-login card, activity preview list of 5 audit events, and the three sanctioned-modal trigger buttons.

### Edge Cases

- **Permission matrix legend**: the role-details page uses **three** distinct icon/text glyphs in matrix cells: ✓ in success-green for «مسموح», ✕ in danger-red for «محجوب», and a small dash (–) in neutral-gray for «غير منطبق» (e.g., the Tax Settings × Approve cell for the Student role — Students don't approve tax). The legend MUST be the topmost element of the matrix block. Color MUST never be the sole indicator of meaning per Constitution IV — text labels accompany each glyph in the legend.
- **Inactive role demonstration**: at least one of the seven roles MUST carry the «موقوفة» status. Per the seeded scenario the «Operations» role is currently «نشطة» — the demonstration of «موقوفة» appears instead on a deprecated **«مساعد إداري»** role (the 8th card on the roles-permissions page) carrying status «موقوفة» with «إعادة تفعيل» CTA replacing the «عرض/تعديل» CTA. This makes the 7 active + 1 archived shape visible, mirroring Spec 009's salary-rule موقوفة pattern (rule 8 «إعادة تفعيل» CTA).
- **User without linked entity**: the super-admin محمد بن عبدالعزيز الإدريسي has no linked student / teacher / family — his user-details page MUST show the «الحساب المرتبط» card with caption «لا يوجد حساب مرتبط — هذا حساب إداري بحت» rather than hiding the card. This communicates that the linkage is structurally always shown (just empty for admin-only accounts), which is more discoverable than conditional rendering.
- **Currency-secondary chip behavior on platform-settings**: the secondary-currency chip group contains EGP as the only seeded chip with a small ✕ button visually present. ### Key Entities

- **Role**: A named bundle of permissions assigned to one or more Users. Attributes: `role_id` (string slug, e.g., `finance_manager`), Arabic name, English subtitle, role type (مدير / تشغيلي / دعم), users count, status (نشطة / موقوفة), 19-module × 7-action permission grid, last-modified timestamp + actor. Seven seeded active roles + one archived («مساعد إداري»).
- **Permission**: A single (Module, Action) pair on a single Role. Module is one of 19 (Dashboard / Students / ... / Audit Log); Action is one of 7 (View / Create / Edit / Delete / Approve / Export / Financial access); value is one of three states (مسموح ✓ / محجوب ✕ / غير منطبق –).
- **User**: A named human account on the platform. Attributes: `user_id`, Arabic name, English transliteration, email, phone (masked), assigned role, status (نشط / موقوف / موثّق / غير موثّق), last-login timestamp + IP (masked), sessions count, 2FA status, linked entity (Student / Teacher / Parent / Family — nullable for admin-only accounts). 14 seeded users.
- **Pricing Rule**: A monetary rule that determines an invoice line-item price. Attributes: `rule_id`, rule type (course-flat / live-group / live-private / Egypt-tier), student type (ناطق / غير ناطق / both), price + currency, applies-to (course-id or session-class), effective date, status (نشطة / موقوفة), last-modified actor + timestamp. 8 active + 1 موقوفة seeded.
- **Student Type**: A platform-wide category that drives pricing-rule-tier and salary-rule-tier selection. Two values locked: «ناطق بالعربية» / «غير ناطق بالعربية» — with four behavior columns (pricing tier reference, teacher salary bucket reference, invoice label, report label).
- **Notification Setting**: A (Category, Channel) toggle pair. Six active categories × four channels = 24 toggles + one deprecated row (تذكير دورة جديدة, 4 toggles all OFF).
- **Audit Event**: A single timestamped record of an admin or system action. Attributes: `event_id`, actor (user_id or "نظام تلقائي"), action verb, module, target identifier (e.g., STL-2026-04-TCH-0042), diff payload (or "—"), timestamp, IP (masked). 24 seeded events spanning 7 days × 11 modules × 6 actors.
- **Platform Setting**: A platform-wide configuration value. Categories: identity (academy name, logo), regional (country, primary + secondary currencies, languages, direction), contact (phone, email, address, website), invoice footer, certificate issuer / signature / seal. Saved-state metadata: last-saved timestamp + actor.
