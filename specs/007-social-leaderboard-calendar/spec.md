# Feature Specification: Social Hub, Leaderboard, and Calendar Frontend

**Feature Branch**: `007-social-leaderboard-calendar`
**Created**: 2026-05-09
**Status**: Draft
**Input**: User description: "Build a safe educational community experience with motivational posts, student recognition, fair ranking, and calendar events. The design must encourage students without creating negative pressure."

## Overview

This feature ships the **Social Hub, Leaderboard, and Calendar** layer that every prior spec has been pointing at since Spec 002. Seven new pages — four on the student side (`social-hub`, `post-details`, `leaderboard`, `calendar`), two on the admin side (`social-hub`, `create-post`), and one on the teacher side (`calendar`) — turn the academy's promise of *safe community + fair recognition + clear scheduling* into a visible, motivation-positive surface.

**Inbound prior-spec links that 404 today** (every link below is wired into a real `<a href>` on a prior-spec page and lands on 404 until Spec 007 ships):

- `pages/student/social-hub.html` — student dashboard sidebar entry (line 114) + the constitutional "Social Hub" sidebar slot on every student page.
- `pages/student/leaderboard.html` — student dashboard sidebar entry (line 122) + the constitutional "Leaderboard" sidebar slot on every student page.
- `pages/student/calendar.html` — student dashboard sidebar entry (line 130) + body CTA (line 474, "عرض الجدول") + the constitutional "Calendar" sidebar slot on every student page.
- `pages/admin/social-hub.html` — admin dashboard sidebar entry (line 183) + the constitutional "Social Hub" admin sidebar slot. _Filename matches the existing inbound link per Q1 — zero prior-spec edits required._
- `pages/teacher/calendar.html` — teacher dashboard sidebar entry (line 103) + body CTA (line 212, "جدولي اليوم") + the constitutional "Calendar" teacher sidebar slot. _Included as a 7th page per Q2, with teacher-specific event data._

Spec 007 adds seven pages:

1. `pages/student/social-hub.html` — the safe educational feed: 7-type post taxonomy (إعلان عام / تهنئة / إنجاز طالب / تحدي الأسبوع / نصيحة تعليمية / تذكير اختبار / منشور دورة), composer-less feed (students consume; only admins author — discovered via `create-post.html`), curated positive-only reactions, comment-counts, "View post" CTA into `post-details.html`, and visible moderation language ("التعليقات تخضع للمراجعة قبل النشر") so the page never feels like an open social network.
2. `pages/student/post-details.html` — single-post deep-read: full post body (image / video placeholder visual, Arabic copy, type badge, author, date, related-course chip when applicable), ≥ 4 sample comments with explicit status badges (تم الاعتماد / في انتظار المراجعة), and a visual-only "أضف تعليقاً" form whose submit button reveals the "سيتم مراجعة تعليقك قبل ظهوره" notice — making moderation *legible* on the page.
3. `pages/student/leaderboard.html` — the motivational ranking board: month selector, "بطاقة ترتيبك" personal-rank card (the persona's own row pulled out before the table — so weaker students still see *their* progress framed individually), six tabs (الترتيب العام / الحضور / الواجبات / الأكثر تطوراً / تقدّم القرآن / المشاركة), and a top-10 table with rank / student / points / badge / progress columns. Tab #4 ("الأكثر تطوراً") explicitly recognises students who climbed the most over the last month — the brief's required fairness affordance for students who are not at the top of the absolute ranking.
4. `pages/student/calendar.html` — the unified events surface: month-grid view (CSS-only, top of page) + an event-list view below for mobile-first scanning, 5-type event taxonomy (حصة مباشرة / موعد تسليم واجب / اختبار / إصدار تقرير / إعلان أكاديمي), 4-state status pills (قادم / مكتمل / فائت / اليوم), and per-event cards with title / date+time / type / status / CTA into the relevant existing prior-spec page (`live-session-details.html` / `assignment-details.html` / `exam-details.html` / `monthly-report.html` / `post-details.html`).
5. `pages/admin/social-hub.html` — the admin moderation workspace: posts table with filters (type / status / date), per-row visual actions (عرض / تعديل / نشر / إلغاء نشر / حذف — visual only), and a moderation summary strip showing pending-comments / published-posts / scheduled-posts counts. The "إنشاء منشور" CTA links to `create-post.html`. _Filename matches the existing admin-dashboard sidebar link per Q1 — zero prior-spec edits required._
6. `pages/admin/create-post.html` — the admin authoring form: post-type select (7 types matching the student feed), title input, content textarea, audience radio group (جميع الطلاب / دورة محددة / أولياء الأمور / المعلمين), upload-image visual placeholder, publish-now / schedule-for-later toggle with a date+time picker visual, and a "حفظ" + "نشر" + "حفظ كمسودة" three-button cluster (visual only).
7. `pages/teacher/calendar.html` — the teacher's unified events surface: same month-grid + event-list structure as the student calendar (5 event types, 4 status states, identical visual language), but rendering teacher-relevant data — the teacher's own scheduled classes, homework deadlines they need to grade, exam-day invigilation slots, student-report release windows, and academy announcements addressed to teachers. _Included per Q2 — closes the teacher-dashboard inbound links (sidebar L103 + body CTA L212)._

The feature is purely visual / frontend. No real authoring, no real comment submission, no real moderation, no real schedule trigger, no real reactions counter increment. The brief's safety mandate ("safe educational community") and motivation mandate ("Most Improved so weaker students can be recognized") are realised by:

1. **Authorship asymmetry**: Students consume; only admins compose. The student-side `social-hub.html` exposes NO post-composer at all — every post in the feed visibly originates from the academy / a teacher / "إدارة الأكاديمية", reinforcing that the surface is a curated newsletter, not an open feed.
2. **Visible moderation**: Every comment on `post-details.html` carries a status badge ("تم الاعتماد" / "في انتظار المراجعة") and the comment-form footer shows "سيتم مراجعة تعليقك قبل ظهوره". Students learn the rule by reading the page.
3. **Curated reactions**: The reaction set is a fixed positive-only quintet (👍 / ⭐ / ❤️ / 🤲 / 👏) — no thumbs-down, no laughing, no anger. Negative or ambivalent emotions are structurally unavailable, eliminating the ratio-pressure dynamic of open social networks.
4. **Fair ranking**: The leaderboard's six tabs deliberately split rank-worthy dimensions so that a single absolute-overall ranking is never the only narrative. The "الأكثر تطوراً" (Most Improved) tab specifically surfaces students who *grew* the most month-over-month — the brief's explicit fairness lever, modeled here as a tab where a student ranked #18 in الترتيب العام can land at #1.
5. **Personal-rank card**: Every student sees their own row pulled out as a hero card above the table on `leaderboard.html`, regardless of rank. A student at rank #34 still gets a personal card showing "ترتيبك: 34 — حصلت على 215 نقطة هذا الشهر — تطوّرت 6 مراكز عن مارس". The personal card always frames movement positively — even a static rank surfaces the points earned this month rather than the gap to rank #1.

Cross-page narrative anchors to the **persona family** locked since Spec 002 and reinforced through Specs 003-006:

- **Primary student**: عبد الرحمن مؤمن (ع.م) — الصف السادس — مسار القرآن الكريم Level 3 — April 2026 figures from Spec 006: 92% attendance, 88% homework, 88% monthly exam, 89% overall, "ممتاز".
- **Sibling**: سارة مؤمن (س.م) — الصف الثالث — "يحتاج إلى دعم".
- **Parent**: ولي أمر الطالب عبد الرحمن.
- **Primary teacher**: الأستاذ أحمد بن عبد الله (Quran).
- **Secondary teacher**: الأستاذة منى سعد (Arabic).
- **Islamic-Studies teacher**: الأستاذ خالد العبدلي.
- **Admin author voice**: إدارة الأكاديمية (used as the author of "إعلان عام" type posts).

In Spec 007 specifically, **the persona's leaderboard rank is rank #6 in الترتيب العام** (mid-pack — deliberately not rank #1) and **rank #2 in الأكثر تطوراً** (climbed 4 places from rank #10 in March). This pinning is critical: it shows the design works for a strong-but-not-top student rather than the cosmetically easy case of putting the persona at #1.

- Q: The "تهنئة" (Congratulations) post on `social-hub.html` (FR-025 + FR-062) celebrates the persona by full name + exact exam score (e.g., "عبد الرحمن مؤمن — 95%"), but the comments-side anonymization rule (FR-015) and leaderboard pseudonym convention (FR-039) both protect student identity. Should achievement posts use full names (always-public), anonymize matching the leaderboard convention (first-name + last-initial), require opt-in consent, or use first-name only? → A: **Anonymize achievement posts.** "تهنئة" / "إنجاز طالب" posts MUST use the first-name + last-initial format (e.g., "عبد الرحمن م.") matching the leaderboard pseudonym convention (FR-039) and the spirit of the comments anonymization rule (FR-015). This preserves celebration while eliminating the comparison-pressure dynamic the brief explicitly wants to avoid. **Internal privacy posture is now consistent across all student-side surfaces**: comments anonymized to role-titled display, leaderboard pseudonymous, achievement-post subjects pseudonymous. The persona's full name "عبد الرحمن مؤمن" still appears in his OWN dashboard / report / personal-card chrome (where it identifies him to himself), but never in posts or rankings visible to other students.
- Q: The leaderboard's "تقدّم القرآن" (Quran Progress) tab is one of 6 ranking dimensions. Should it rank all students (including Arabic-only / Islamic-Studies-only learners — who would appear with 0 progress, which is punitive and inaccurate), only Quran-enrolled students, all-students-with-non-enrolled-hidden, or be dropped entirely? → A: **Quran-enrolled only.** The "تقدّم القرآن" table MUST render only students currently enrolled in any Quran course. A small caption above the table MUST explain the scope: "هذا الترتيب يشمل الطلاب المسجّلين في دورات القرآن — نقيس التقدّم في المسار، ليس الالتحاق به." This avoids the punitive 0-score case for Arabic-only / Islamic-Studies-only learners (e.g., the sibling سارة, enrolled in Arabic L1, MUST NOT appear in this tab's table) and keeps the dimension meaningful. The persona (enrolled in Quran L3) appears in this tab. The same enrollment-scoping principle applies to any future course-specific dimension. Other dimensions (الترتيب العام / الحضور / الواجبات / الأكثر تطوراً / المشاركة) remain academy-wide.
- Q: The leaderboard surfaces "النقاط" per student across 6 dimensions but the spec doesn't tell the student how points are computed. For the "fair ranking" mandate, should the page expose the calculation formula (transparency builds trust), expose only a per-tab tooltip, expose a static one-line caption, or stay opaque? → A: **Inline accordion explainer.** The page MUST render a single "ⓘ كيف نحسب النقاط؟" info-link below the tabs that — when conceptually expanded — reveals an inline accordion-style explainer with a one-line formula per dimension: "الترتيب العام: مجموع نقاط كل الفئات. الحضور: 5 نقاط لكل حصة حضرت. الواجبات: 10 نقاط لكل واجب مقبول. الأكثر تطوراً: عدد المراكز التي صعدتها مقارنة بالشهر الماضي. تقدّم القرآن: نسبة إكمال محتوى المستوى. المشاركة: تفاعل في الحصص + التعليقات المعتمدة." For the static prototype, the accordion content MAY be rendered open by default (to make it visible without JS), or wired to the existing `data-accordion`-style toggle if such a handler exists in `assets/js/main.js`. The explainer is the page's visible answer to the trust question every student silently asks.
- Q: The spec mandates a "في انتظار المراجعة" status badge (FR-013) and a moderation-helper line on the comment form (FR-014), but doesn't disclose a moderation-timeline expectation. Without one, a commenter who doesn't see their comment appear may lose trust. Should the page surface a 24-hour SLA, a 48-hour SLA, vague language, or no SLA copy? → A: **24-hour SLA.** FR-014's helper text MUST be extended to read "يرجى الالتزام بآداب الحوار التعليمي. التعليقات تخضع للمراجعة قبل النشر — نراجع التعليقات عادةً خلال 24 ساعة." The "في انتظار المراجعة" status badge MUST also carry a tooltip surfacing the same 24-hour expectation ("سيتم نشر هذا التعليق بعد المراجعة — عادةً خلال 24 ساعة.") so the timeline is visible at both submission time AND status-display time. The 24-hour window aligns with educational-platform norms and sets concrete trust-positive expectations without overcommitting to a real production SLA.
- Q: The spec mandates positive-only reactions with per-reaction counts (FR-012) but doesn't specify what happens when a student taps a reaction button. Three meaningfully different behaviors are possible: (a) tap opens a modal listing reactors by full name (re-introduces the identity exposure that the Q1 anonymization decision eliminates), (b) tap opens a modal showing role-titled aggregate counts only ("12 طالباً + 3 معلمين"), or (c) tap is non-interactive (visual highlight only, no reveal). → A: **Non-interactive.** Tapping a reaction button MUST NOT reveal the identities of reactors — neither by name nor by role-titled aggregate. Tap behavior is limited to a momentary visual highlight (CSS-only `:active` / `:focus` styling) with no overlay, modal, tooltip, or popover surfacing reactor identities. This preserves privacy uniformly across all student-side surfaces (matching the Q1 anonymization decision and FR-015) AND matches the static-prototype no-JS posture (no JS handler is needed). Combined with FR-011 (no composer), FR-012 (curated reactions), FR-015 (anonymized post + comment subjects), and FR-014 (visible moderation + 24h SLA), this closes the last back-door identity-reveal vector and completes the "no open social network feel" mandate.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Opening the Social Hub Feed (Priority: P1) 🎯 MVP

A student opens the **Social Hub** from the sidebar (or from the dashboard's "آخر منشورات الأكاديمية" CTA). They want a calm, scannable feed of academy-curated posts: announcements, congratulations to students, weekly challenges, educational tips, exam reminders, and course-specific posts — visibly authored by the academy or named teachers, with positive-only reactions, comment counts, and a clear "View post" path. They must NOT find an open compose box, an emoji of laughter / anger / thumbs-down, or any UI element that resembles a public social network's "post status" affordance.

**Why this priority**: This is the canonical Social Hub surface — the entry point three sidebar/dashboard inbound links already target. Without it, the constitution's "Social Hub" sidebar entry on every student page is a dead end. P1 because it is the brief's headline page and the "safe community" promise has no other home.

**Independent Test**: Open `pages/student/social-hub.html` directly from the file explorer. Without any other context, the viewer must be able to answer in 30 seconds: "Whose academy is this? Who authored each post? When? What type of post is each one (announcement / congratulations / challenge / tip / exam reminder / course post)? Is there a compose-your-own box (no — students consume only)? Are reactions limited to a positive-only set? Are comments visibly moderated? Where do I open a post in full?" — entirely from the page.

**Acceptance Scenarios**:

1. **Given** the student opens social-hub, **When** they look at the top, **Then** they see h1 "مجتمع إدارتي" + a one-line subtitle ("مساحة تعليمية آمنة — منشورات الأكاديمية، تهانينا، وتحديات أسبوعية") and a small explainer chip below the title ("التعليقات تخضع للمراجعة — هذا فضاء تعليمي، ليس شبكة اجتماعية مفتوحة") rendered as a soft info-tinted banner with a shield icon.
2. **Given** the page enforces authorship asymmetry, **When** the student scans the page, **Then** they DO NOT see any "اكتب منشوراً" / "ما الذي يدور في ذهنك؟" / composer textarea anywhere on the page. The only authoring affordance is in the sidebar / nowhere on this surface.
3. **Given** the student wants the feed, **When** they scroll the main column, **Then** they see ≥ 7 post cards rendered in reverse-chronological order, one card per post type from the 7-type taxonomy: (1) "إعلان عام" — academy-authored holiday/closure announcement; (2) "تهنئة" — congratulating "عبد الرحمن م." (anonymized first-name + last-initial per FR-015 + the achievement-post privacy rule) on passing the March monthly exam at 95%; (3) "إنجاز طالب" — celebrating "سارة م." (or a third anonymized name) on completing a level; (4) "تحدي الأسبوع" — weekly Quran-recitation challenge from الأستاذ أحمد; (5) "نصيحة تعليمية" — study-skill tip from الأستاذة منى سعد; (6) "تذكير اختبار" — May 28 monthly-exam reminder; (7) "منشور دورة" — course-specific post tied to "القرآن الكريم — المستوى الثالث".
4. **Given** each post must read as authored, **When** the student looks at any card, **Then** they see a structured header row with: author avatar (circular), author name (chip — "إدارة الأكاديمية" / "الأستاذ أحمد بن عبد الله" / "الأستاذة منى سعد" / "الأستاذ خالد العبدلي"), date (relative — "منذ ساعتين" or absolute — "5 مايو 2026"), and a type-pill rendered with a distinct color + icon per type (7 types → 7 distinct color+icon pairs, never color alone).
5. **Given** each post needs visual richness, **When** the student looks at a post body, **Then** they see: a 2-3 sentence Arabic content paragraph, an image placeholder visual (aspect-ratio 16:9, soft border, illustrative rather than photographic — e.g., the academy crest, a Quran icon, a calendar visual) with descriptive `alt` text, and ≥ 1 line of meta below the image where applicable (e.g., "مرتبط بدورة: القرآن الكريم — المستوى الثالث" with a chip linking to `course-details.html`).
6. **Given** the reactions must be positive-only, **When** the student looks at the post footer, **Then** they see exactly 5 fixed reaction buttons (👍 إعجاب / ⭐ تميّز / ❤️ محبّة / 🤲 دعاء / 👏 تشجيع) with per-reaction counts; thumbs-down / laughing / anger reactions MUST NOT appear anywhere on the page. Below the reactions: a comments-count chip ("12 تعليقاً") and a primary "عرض المنشور" CTA → `post-details.html`.
7. **Given** the page is filterable by type, **When** the student looks above the feed, **Then** they see a filter chip strip with 8 chips ("الكل" + 7 type chips); the "الكل" chip is selected by default. The strip is visual-only.
8. **Given** the page is responsive, **When** the student opens at mobile width (< 768 px), **Then** the post cards stack to a single column at full container width, the type-filter chips become horizontally scrollable rather than wrapping, and image placeholders preserve their 16:9 ratio with no horizontal overflow.

### User Story 2 — Reading a Single Post and Its Moderated Comments (Priority: P1)

A student taps "عرض المنشور" on any social-hub feed card and lands on the **Post Details** page. They want the full post (longer content, full image, related-course chip if applicable), the comments thread with explicit moderation-status badges on every comment, and a visual-only "Add comment" form whose submit copy makes the moderation rule unmistakable.

**Why this priority**: This is the only page on which the safety mandate becomes operationally visible. The social-hub feed *implies* moderation; this page *shows* it via per-comment status badges and pre-submit copy. Without this page, the brief's "Comments appear moderated visually" requirement has no visual surface. P1 alongside Story 1.

**Independent Test**: Open `pages/student/post-details.html` directly. A viewer must be able to answer: "What is this post about? Who authored it? When? Is it tied to a specific course? Are there comments — how many — and which are approved vs awaiting review? Does the comment form make moderation visible (i.e., does it tell me that my comment will be reviewed before it appears)?" — entirely from the page.

**Acceptance Scenarios**:

1. **Given** the student opens post-details, **When** they look at the top, **Then** they see a breadcrumb (مجتمع إدارتي / تفاصيل المنشور), the post-type pill (e.g., "تحدي الأسبوع" — accent-100 with star icon), and the post title h1 ("تحدي الأسبوع: راجع سورة الفجر مع تطبيق أحكام التجويد").
2. **Given** the student wants the post body, **When** they look below the title, **Then** they see: author strip (avatar + name "الأستاذ أحمد بن عبد الله" + date "5 مايو 2026"), an optional related-course chip ("مرتبط بدورة: القرآن الكريم — المستوى الثالث" → `course-details.html` for posts of type "تحدي الأسبوع" / "منشور دورة" / "تذكير اختبار" / "تهنئة"; or `#planned` for "إعلان عام" / "نصيحة تعليمية" / "إنجاز طالب" with a "غير مرتبط بدورة محددة" caption instead), a hero image placeholder (aspect-ratio 16:9), and a 4-6 paragraph Arabic body with appropriate line-height (≥ 1.7) and font size (≥ 16 px).
3. **Given** the post supports positive-only reactions, **When** the student looks at the reactions row, **Then** they see the same 5-reaction quintet from Story 1 with per-reaction counts (e.g., 👍 14 / ⭐ 9 / ❤️ 7 / 🤲 4 / 👏 11) and a comments-count chip ("8 تعليقات").
4. **Given** the comments must visibly show moderation status, **When** the student scrolls to the comments section, **Then** they see ≥ 4 comment items with: commenter avatar, commenter display name (anonymized — "طالب من المستوى الثالث" / "ولي أمر" / "طالبة من المستوى الثاني" — full names are NOT shown), date, comment body (1-3 sentences each), and a status badge per comment with exactly two states: "تم الاعتماد" (success-100 + check icon — visible on ≥ 2 comments) and "في انتظار المراجعة" (warning-100 + clock icon — visible on ≥ 1 comment, with a muted body styling that signals "this will appear publicly only after approval"). At least one comment per status MUST be present so the badge vocabulary is visibly demonstrated.
5. **Given** the comment form must make moderation legible, **When** the student looks at the "أضف تعليقاً" form, **Then** they see: a textarea with placeholder ("اكتب تعليقاً ملائماً ومحترماً..."), a helper text below the textarea ("يرجى الالتزام بآداب الحوار التعليمي. التعليقات تخضع للمراجعة قبل النشر — نراجع التعليقات عادةً خلال 24 ساعة.") rendered in info-tinted style with a shield icon, and a primary "إرسال للمراجعة" button (NOT "نشر" / "Publish" — the verb structurally encodes that the comment will be reviewed). A small inline notice clarifies the prototype-only nature ("هذا نموذج تجريبي — لا يتم إرسال تعليقات فعلية."). The "في انتظار المراجعة" badges in the comments thread above carry a tooltip surfacing the same 24-hour SLA so the moderation timeline is visible from two angles on the page.
6. **Given** the post may not exist in isolation, **When** the student looks at the bottom, **Then** they see a "منشورات أخرى قد تهمك" panel with ≥ 3 related-post mini-cards (each linking back into `social-hub.html` or `post-details.html` with a different sample post pre-filled in the prototype).
7. **Given** the page is responsive, **When** the student opens at mobile width, **Then** the comments stack to single column, the comment form's textarea remains at least 4 rows tall, and the related-posts panel collapses to a horizontally scrollable strip.

### User Story 3 — Browsing the Leaderboard (Priority: P1)

A student opens the **Leaderboard** from the sidebar. They want a motivational ranking surface with a month selector at the top, a personal-rank "بطاقة ترتيبك" card pulled out so they can see *their own* progress before the table even loads, six rank-type tabs (الترتيب العام / الحضور / الواجبات / الأكثر تطوراً / تقدّم القرآن / المشاركة), and a top-10 table with rank / student / points / badge / progress columns. Crucially, the page must *not* trigger negative pressure: no "you are far from the top" framing, and the "الأكثر تطوراً" tab must give weaker students a path to recognition.

**Why this priority**: This is the brief's most tonally-sensitive page. The "fair ranking" + "Most Improved" requirement explicitly addresses the risk that a single absolute ranking demotivates lower-ranked students. P1 because: (a) the sidebar entry "Leaderboard" is constitutionally guaranteed on every student page; (b) the page is the proof-point of the brief's "encourage students without creating negative pressure" objective.

**Independent Test**: Open `pages/student/leaderboard.html` directly. A viewer must be able to answer: "Which month is being ranked? What is the persona's own rank, points, and movement-direction? What rank dimensions are available (overall / attendance / homework / most-improved / Quran progress / participation)? Who are the top-10 students in the currently-selected dimension? ### Edge Cases

### Key Entities

- **Post**: A single feed item authored by an admin or teacher. Attributes: type (one of 7), title, body (multi-paragraph Arabic), author (admin or teacher persona), date, optional related-course, image placeholder, audience (one of 4), status (one of 5: منشور / مسودة / مجدول / غير منشور / محذوف), reaction-counts-by-emoji, comments-count.
- **Comment**: An anonymized opinion on a post. Attributes: anonymized author display, comment body, date, status (one of 2: تم الاعتماد / في انتظار المراجعة).
- **Reaction**: A positive-only emoji + count pair. Fixed quintet: 👍 إعجاب / ⭐ تميّز / ❤️ محبّة / 🤲 دعاء / 👏 تشجيع. Each post has 5 such pairs.
- **Leaderboard rank entry**: A student's ranking in a single dimension for a single month. Attributes: student (avatar + display name), rank (1-N), points, badge label, progress %, month-over-month movement.
- **Leaderboard dimension**: One of 6 ranking lenses: الترتيب العام / الحضور / الواجبات / الأكثر تطوراً / تقدّم القرآن / المشاركة.
- **Calendar event**: An entry in the academy's monthly schedule. Attributes: title, date, time, type (one of 5: حصة مباشرة / تسليم واجب / اختبار / إصدار تقرير / إعلان أكاديمي), status (one of 4: قادم / مكتمل / فائت / اليوم), CTA destination (one of 5 prior-spec page filenames).
- **Audience**: One of 4 admin-side broadcast targets: جميع الطلاب / دورة محددة / أولياء الأمور / المعلمين.
- **Moderation summary**: A 4-card aggregate visible on `social-hub.html`: pending-comments / published-posts / scheduled-posts / drafts.
